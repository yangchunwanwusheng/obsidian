# 端到端研究流水线

> 从"我想研究 XXX"到"论文发表"的完整自动化流程指南。本文档是整个科研工作流的核心路线图，定义了每个阶段的目标、触发条件、自动化步骤和检查清单。

## 流程总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                     端到端研究流水线                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Phase 0: 方向萌芽（1-2周）                                         │
│    ↓ 用户选定方向                                                   │
│  Phase 1: 第一轮精读（2-3周）— 广度扫描 + 深度聚焦                   │
│    ↓ 调研完成                                                       │
│  Phase 2: 方向确定（1周）— Gap Analysis + 研究问题凝练               │
│    ↓ 方向锁定                                                       │
│  Phase 3: 第二轮精读（2-3周）— 针对性深度调研 + 基线复现              │
│    ↓ 基线跑通                                                       │
│  Phase 4: 项目规划（1周）— 实验方案设计 + 资源预算                    │
│    ↓ 规划完成                                                       │
│  Phase 5: 实验执行（4-8周）— 日常实验 + 日志记录                     │
│    ↓ 结果充分                                                       │
│  Phase 6: 论文写作（3-5周）— 逐章节撰写 + 润色                      │
│    ↓ 初稿完成                                                       │
│  Phase 7: 投稿（2-4周+）— 选刊 + 格式调整 + 提交                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**时间线参考**：整个流程约 16-30 周（4-8 个月），具体取决于方向的成熟度和实验的复杂度。

## Phase 0：方向萌芽

**触发条件**：用户说"我想研究 XXX"、"帮我找论文选题"、"XXX 方向现在还能做什么"

**时间估计**：1-2 周

### 目标

从模糊的研究兴趣出发，识别 3-5 个有潜力的候选方向，供用户选择。

### 自动化步骤

1. **前沿扫描**
   - Claude Code 调用 Semantic Scholar API 搜索前沿方向
   - 搜索近 1-2 年顶会论文（CVPR / ICCV / NeurIPS / ICML / ACL / AAAI）
   - 识别热门子方向（按论文数量和增长率排序）
   - 重点关注论文的 "Future Work" 章节——这是作者验证过的可行方向

2. **初步可行性评估**
   - 对每个子方向评估：
     - **算力需求**（高/中/低）——是否需要多卡训练
     - **数据集获取难度** ——公开可用/需申请/需自建
     - **代码复杂度** ——从零实现/基于开源工具箱修改
     - **创新空间** ——饱和/有缺口/新兴蓝海
   - 结合用户背景（研究者身份档案中的技能树）给出匹配度评分

3. **候选方向列表生成**
   - 为每个候选方向创建 `research/01-topics/[方向].md`
   - 每个方向包含：
     - 方向概述（2-3 句话）
     - 核心研究问题
     - 创新空间分析
     - 可行性评估（算力/数据/时间）
     - 3-5 篇代表性论文
     - 初步风险评估

4. **交付用户决策**
   - 产出 3-5 个候选方向列表
   - 用户选择 1-2 个方向进入 Phase 1

### 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| 方向笔记 | `research/01-topics/[方向].md` | 每个候选方向一份 |
| 论文列表 | 嵌入方向笔记中 | 代表性论文的标题/链接/摘要 |

### 检查清单

→ [[research/09-workflow/phase-checklists/01-topic-selection|选题检查清单]]

## Phase 1：第一轮精读

**触发条件**：用户从 Phase 0 选定 1-2 个方向

**时间估计**：2-3 周

### 目标

理解领域全貌，识别主要流派和技术路线，为方向确定提供知识基础。

### 两轮调研策略

本阶段采用"两轮调研"而非一次性阅读，确保既有广度又有深度：

#### 第一轮：广度扫描

**目标**：建立领域全景图，理解主要技术流派

**自动化步骤**：
1. 搜索 20-30 篇相关论文（Semantic Scholar + arXiv）
2. 自动创建 reading-queue（按相关度排序）
3. 对每篇论文快速阅读：
   - Abstract（核心贡献一句话）
   - Conclusion（主要结论和 Future Work）
   - Method Overview（方法架构概览，不深入数学细节）
4. 使用 ChatPaper / Elicit / SciSpace 快速获取结构化摘要
5. 按技术流派分类，归入 `research/02-papers/` 子目录

**分类维度示例**：
| 维度 | 示例 |
|------|------|
| 技术路线 | Transformer-based / CNN-based / Hybrid |
| 问题类型 | 分类 / 检测 / 分割 / 生成 |
| 数据模态 | 图像 / 视频 / 多模态 |
| 性能导向 | 精度优先 / 速度优先 / 轻量化 |

#### 第二轮：深度聚焦

**目标**：锁定 3-5 篇最相关的核心论文，理解技术细节

**自动化步骤**：
1. 从第一轮筛选出 3-5 篇核心论文
2. 精读每篇论文（使用 paper-note 模板）：
   - 完整的方法理解（公式逐符号解释）
   - 实验设置分析（数据集/基线/指标）
   - 优点和局限性
3. 分析各方法之间的关系和差异
4. 识别研究空白（Gap）
5. 自动生成论文关系图（引用网络可视化）

**精读笔记模板结构**：
```markdown
# 论文标题
## 核心贡献（一句话）
## 方法详解（逐步拆解）
## 实验分析
## 局限性
## 与其他方法的关系
## 潜在改进方向（我的思考）
```

### 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| Reading Queue | `research/02-papers/reading-queue.md` | 按优先级排序的论文列表 |
| 论文精读笔记 | `research/02-papers/[论文].md` | 核心论文的完整笔记 |
| 论文关系图 | `research/02-papers/[方向]-knowledge-graph.md` | 方法间的关系和演进 |
| 流派分类表 | 嵌入方向笔记中 | 技术路线对比 |

## Phase 2：方向确定

**触发条件**：Phase 1 两轮调研完成后

**时间估计**：1 周

### 目标

基于充分的文献调研，凝练出 3 个具体可执行的研究问题，由用户最终选定一个。

### 核心原则

**给出方向之前，必须对该方向做两轮调研分析，判断创新性是否存在、是否还可以发论文。** 绝不能在调研不充分的情况下推荐方向。

### 自动化步骤

1. **Gap Analysis（研究空白分析）**
   - Claude Code 综合所有精读笔记，分析现有工作的局限性
   - 识别三类研究空白：
     - **问题空白**：尚未被研究的问题
     - **方法空白**：现有方法的不足之处
     - **场景空白**：现有方法未覆盖的应用场景
   - 对每个空白评估：重要性 / 可行性 / 创新空间

2. **候选研究问题生成**
   - 给出 3 个具体研究问题，每个包含：
     - **具体研究问题**（一句话描述）
     - **核心创新点描述**（与现有工作的本质区别）
     - **可行性评估**（算力/数据/时间/代码复杂度）
     - **风险评估**（可能失败的原因和应对策略）
     - **预期贡献**（理论贡献/实践贡献）
     - **3-5 篇核心参考文献**（最相关的基线论文）
     - **预计可投的会议/期刊**

3. **用户决策**
   - 用户选择最终方向
   - 更新 `research/01-topics/[方向].md` 的状态为"选定"
   - 记录到科研日志

### 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| Gap Analysis 报告 | 嵌入方向笔记中 | 研究空白识别结果 |
| 研究问题列表 | `research/01-topics/[方向].md` | 3 个候选研究问题 |
| 科研日志 | `research/07-journal/daily/` | 记录方向确定过程 |

### 关键决策点

```
用户选择研究问题
  → 是否有足够创新性？ → 否 → 回到 Phase 1 补充调研
  → 是否有足够资源？   → 否 → 调整问题范围或换方向
  → 两者都满足         → 进入 Phase 3
```

## Phase 3：第二轮精读

**触发条件**：方向确定后（Phase 2 完成）

**时间估计**：2-3 周

### 目标

针对选定方向进行深度调研，理解最前沿的技术细节，复现基线代码，为方法设计奠定基础。

### 与 Phase 1 的区别

| 维度 | Phase 1 | Phase 3 |
|------|---------|---------|
| 范围 | 广泛了解领域全貌 | 聚焦选定方向 |
| 论文数 | 20-30 篇扫描 | 30-50 篇搜索，5-10 篇精读 |
| 深度 | 方法概览 | 公式级理解 + 代码级复现 |
| 目标 | 识别研究方向 | 为方法设计打基础 |

### 自动化步骤

1. **针对性深度搜索**
   - 针对选定方向搜索 30-50 篇论文
   - 重点搜索：
     - 核心基线论文及其后续改进
     - 同一问题不同解法的论文
     - 相关工具/数据集论文
   - 创建精细化的 reading-queue

2. **核心基线精读（5-10 篇）**
   - 对每篇核心论文做公式级精读
   - 分析代码实现（如果开源）
   - 理解训练细节（超参数、数据增强、trick）
   - 复现关键实验结果

3. **基线代码复现**
   - Claude Code 辅助复现基线代码
   - 使用 OpenMMLab / Detectron2 等开源工具箱
   - 在复现过程中发现问题：哪些地方可以改进？
   - 记录复现中的观察和想法

4. **改进想法收集**
   - 产出精读笔记 + 代码复现 + 改进想法
   - 改进想法记录到 `research/_inbox/`
   - 每个想法包含：
     - 灵感来源（哪篇论文/哪个实验结果）
     - 具体描述（想做什么改进）
     - 预期效果（希望达到什么目标）
     - 初步可行性评估

### 论文材料记录

所有与论文相关的关键发现、创新灵感都需记录。这些记录是后续论文写作的素材库：

| 记录类型 | 说明 | 存放位置 |
|----------|------|---------|
| 实验观察 | 训练曲线、loss 分析、bad case | `research/03-experiments/` |
| 技术灵感 | "如果把 A 和 B 结合会怎样" | `research/_inbox/` |
| 关键数据 | SOTA 对比数据集的指标 | 论文笔记中 |
| 创新点雏形 | 潜在的方法创新思路 | `research/_inbox/` |

### 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| 深度精读笔记 | `research/02-papers/[论文].md` | 5-10 篇核心论文 |
| 基线代码 | 本地或 Kaggle | 可运行的基线实现 |
| 改进想法集 | `research/_inbox/` | 收集的灵感和想法 |
| 复现报告 | `research/03-experiments/` | 基线复现结果 |

## Phase 4：项目规划

**触发条件**：基线复现完成，有明确改进方向（Phase 3 完成）

**时间估计**：1 周

### 目标

将模糊的改进想法转化为可执行的实验计划，明确资源需求和里程碑。

### 自动化步骤

1. **实验方案设计**
   - Claude Code 辅助制定实验方案
   - 基于改进想法集（`research/_inbox/`）和基线复现经验
   - 确定核心创新点的实验验证策略

2. **实验配置确定**
   - **数据集选择**：根据研究问题选择合适的数据集
   - **基线方法**：确定对比的 SOTA 方法（3-5 个）
   - **评估指标**：主指标 + 辅助指标
   - **实验环境**：Kaggle / Colab / 本地服务器

3. **实验设计**
   - **主实验**：与 SOTA 对比，验证方法有效性
   - **消融实验**：验证每个模块的贡献
   - **分析实验**：可视化、case study、失败分析
   - **效率实验**：速度/内存/参数量对比

4. **资源预算**
   - GPU 时间估算（每轮实验的时长 x 实验轮数）
   - API 费用预算（如使用付费 API）
   - 时间节点规划（里程碑 + deadline）

5. **计划文档输出**
   - 创建 `research/03-experiments/` 下的实验计划
   - 输出使用 WPS 可编辑的格式（.docx 或 .md）

### 实验计划模板

```markdown
# 实验计划：[研究问题]

## 研究目标
## 核心创新点
## 实验配置
  - 数据集：...
  - 基线方法：...
  - 评估指标：...
  - 实验环境：...

## 实验列表
### Exp-1: 主实验（对比 SOTA）
  - 目的：...
  - 设置：...
  - 预期结果：...

### Exp-2: 消融实验
  - 目的：...
  - 设置：...
  - 预期结果：...

## 时间规划
| 阶段 | 时间 | 内容 |
|------|------|------|
| 第1周 | ... | ... |

## 风险预案
- 如果 A 失败 → 应对策略 B
```

### 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| 实验计划 | `research/03-experiments/experiment-plan.md` | 完整实验方案 |
| 甘特图/时间线 | 嵌入计划中 | 里程碑和 deadline |
| 资源预算表 | 嵌入计划中 | GPU/API 费用估算 |

## Phase 5：实验执行

**触发条件**：项目规划完成（Phase 4 完成）

**时间估计**：4-8 周（整个流程中最耗时的阶段）

### 目标

按照实验计划执行实验，记录过程和结果，积累论文素材。

### 日常流程

```
每日循环：
  1. 查看今日实验目标
  2. 编写/修改实验代码
  3. 提交训练任务
  4. 等待结果，分析输出
  5. 记录实验日志
  6. 记录论文材料（关键发现）
  7. 调整下一步计划
```

### 两类记录（核心区分）

本阶段必须区分两类记录，它们服务于不同目的：

#### 1. 日志记录（过程追踪）

**目的**：记录"做了什么、为什么做、有什么用"

| 内容 | 说明 | 存放位置 |
|------|------|---------|
| 日常实验日志 | 今天做了什么实验、结果如何 | `research/07-journal/daily/` |
| 周报 | 本周进展总结、下周计划 | `research/07-journal/weekly/` |
| 代码变更记录 | 修改了哪些代码、为什么 | Git commit message |

**日志记录原则**：
- 即使实验失败也要记录（失败的实验同样有价值）
- 记录决策理由（为什么选择 A 而不是 B）
- 记录时间花费（方便后续估算）

#### 2. 论文材料记录（内容积累）

**目的**：积累可凝练为论文内容的关键发现

| 内容 | 说明 | 存放位置 |
|------|------|---------|
| 关键数据 | 实验结果表格、数值 | `research/03-experiments/` |
| 创新点验证 | 创新方法有效的证据 | `research/03-experiments/` |
| 可视化图表 | 训练曲线、对比图、case study | `research/03-experiments/figures/` |
| 消融分析 | 每个模块的贡献数据 | `research/03-experiments/` |

**论文材料记录原则**：
- 只记录"能写进论文"的内容
- 数据要完整（包含设置、结果、分析）
- 图表要清晰（可直接用于论文）

### 自动化辅助

- **Claude Code 辅助**：代码实现、调试、结果分析
- **每日日志**：用户说"记录今天实验" → 自动填充 experiment-log 模板
- **周报生成**：每周末自动汇总本周日志 → `research/07-journal/weekly/`
- **关键发现同步**：重要结果及时更新到方向文档和论文笔记

### 算力优化策略

| 策略 | 说明 |
|------|------|
| 预训练 + 微调 | 不要从零训练，基于预训练权重微调 |
| 小数据验证 | 先在数据子集上快速验证想法 |
| 渐进式扩展 | 小实验验证 → 中等规模确认 → 全量训练 |
| 超参搜索 | 先粗搜索再细搜索，避免盲目调参 |
| 早停策略 | 设置合理的 early stopping，避免过拟合 |

### 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| 实验日志 | `research/07-journal/daily/` | 每日实验记录 |
| 周报 | `research/07-journal/weekly/` | 每周进展总结 |
| 实验结果 | `research/03-experiments/` | 完整实验数据 |
| 可视化图表 | `research/03-experiments/figures/` | 论文级图表 |
| 科研日志 | `research/07-journal/daily/` | 综合科研记录 |

## Phase 6：论文写作

**触发条件**：实验完成，有足够结果支撑论文

**时间估计**：3-5 周

### 目标

将实验结果组织为结构完整的学术论文初稿。

### 写作流程

```
Step 1: 确定论文框架（1-2天）
  → 核心贡献点（3个）
  → 叙事主线
  → 目标会议/期刊

Step 2: 分章节写作（2-3周）
  Abstract → Introduction → Related Work → Method → Experiment → Conclusion

Step 3: 图表制作（3-5天）
  → 方法框架图、实验对比表、消融分析图

Step 4: 润色和格式化（3-5天）
  → 语言润色、格式调整、参考文献整理
```

### 自动化步骤

1. **论文框架创建**
   - 使用 paper-writing 模板创建写作框架
   - Claude Code 辅助确定 3 个核心贡献点
   - 规划叙事结构："问题引入 → 现有工作不足 → 我们的方法 → 实验验证"

2. **逐章节写作**
   - Claude Code 辅助逐章节写作
   - 写作顺序：Method → Experiment → Introduction → Related Work → Abstract → Conclusion
   - 每章写作流程：
     - 用户提供核心要点和实验数据
     - Claude Code 扩写为完整章节
     - 用户审阅并修改
   - 用户说"写 Introduction" → 触发 Introduction 写作
   - 用户说"写 Related Work" → 触发 Related Work 写作

3. **语言润色**
   - 使用 GPT Academic / Writefull 润色
   - Claude Code 辅助检查学术表达准确性
   - 用户说"润色 [文件]" → 触发润色流程

4. **图表制作**
   - 工具组合：draw.io（框架图）+ matplotlib/seaborn（数据图）+ detikzify（LaTeX 图）
   - 图表要有自明性：只看图表能理解核心信息
   - 配色方案：统一使用论文级配色

5. **格式转换**
   - 输出 .md → pandoc → .docx（WPS 编辑）或 .tex（LaTeX 排版）
   - 用户说"转成 docx" → 触发格式转换

### 学术语言规范

**必须遵守**：

| 规范 | 错误示例 | 正确示例 |
|------|---------|---------|
| 避免口语化 | "This is good" | "This approach demonstrates superior performance" |
| 避免绝对化 | "we solve the problem" | "we propose a method that alleviates..." |
| 使用谦逊表达 | "our method is the best" | "our method achieves competitive results" |
| 精确描述 | "better results" | "improves mAP by 2.3% on COCO" |

**顶会常用表达**：
- 贡献："we introduce / propose / present / develop"
- 对比："outperforms / surpasses / exceeds / achieves state-of-the-art"
- 局限："while effective, our method has limitations in..."
- 分析："we attribute this to / this suggests that / one possible explanation is"

## Phase 7：投稿

**触发条件**：论文初稿完成（Phase 6 完成）

**时间估计**：2-4 周+（含审稿等待时间）

### 目标

选择合适的目标期刊/会议，完成格式调整和投稿。

### 自动化步骤

1. **选刊/选会**
   - 查阅 `research/08-strategy/journal-registry` 选择目标期刊/会议
   - 考虑因素：
     - 方向匹配度
     - 影响因子/CCF 等级
     - 审稿周期
     - 录用率
     - Open Access 政策
   - 推荐 3 个目标（冲刺/适中/保底）

2. **格式调整**
   - 检查目标会议/期刊的格式要求（页数、模板、引用风格）
   - 调整论文格式
   - 检查参考文献格式一致性
   - 生成 PDF，检查图表清晰度

3. **查重**（如需要）
   - 使用 iThenticate / Turnitin 查重
   - 确保相似度在合理范围内

4. **提交投稿**
   - 按要求准备投稿材料
   - 提交到投稿系统
   - 记录到 `research/05-submission/`

5. **审稿意见处理**
   - 收到审稿意见后使用 review-record 模板记录
   - Claude Code 辅助逐条分析审稿意见
   - 辅助生成回复："感谢审稿人的建议，我们做了以下修改..."
   - 用户决策：是否接受修改、如何在论文中体现

6. **迭代投稿**
   - 如果被拒，分析审稿意见，修改论文
   - 投下一个会议（会议周期参考：CVPR 3 月截稿/11 月出结果）
   - 或转为投 Journal（周期更长但相对宽松）

|------|------|------|
| 投稿记录 | `research/05-submission/[会议-年份].md` | 投稿详情 |
| 审稿意见 | `research/05-submission/[会议-年份]-reviews.md` | 审稿意见记录 |
| 回复信 | `research/05-submission/[会议-年份]-response.md` | 审稿意见回复 |

## 阶段间流转规则

```
Phase 0 → Phase 1：用户选定 1-2 个方向
Phase 1 → Phase 2：两轮调研完成，有足够知识基础
Phase 2 → Phase 3：用户选定研究问题
Phase 3 → Phase 4：基线复现完成，有明确改进方向
Phase 4 → Phase 5：实验计划确定
Phase 5 → Phase 6：实验结果充分，可支撑论文
Phase 6 → Phase 7：论文初稿完成

失败回退：
  Phase 2 失败 → 回到 Phase 1 补充调研
  Phase 3 失败 → 回到 Phase 2 重新选方向
  Phase 5 结果不理想 → 回到 Phase 3 重新审视方法
  Phase 7 被拒 → 回到 Phase 6 修改论文
```
