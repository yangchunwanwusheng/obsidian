# 三类研究策略

> 本科生做研究，与其和博士拼"大而全"，不如在"小而精"上做到极致。以下三类策略各有适用场景，关键是匹配自身条件。

## 策略 A：补上别人没做透的分析

### 定义

找到一篇或多篇已有论文，它们提出了一个有意思的方法但分析不够深入——可能是消融实验不够充分、没有在足够多样的场景下测试、或者对某个关键现象缺少解释。你补上这些缺失的分析，产出一篇"更全面的理解"。

**核心思想**：别人的方法已经验证了"能用"，你来回答"为什么能用"和"什么时候不能用"。

### 适用场景

- 你发现某篇近期论文方法有趣，但实验部分明显有缺漏
- 你对某个已有方法有新的理解角度（比如从信息论/优化/几何角度重新解释）
- 领域内有多个方法在做类似的事，但没人系统对比过它们各自的优缺点和适用范围

### 经典案例

| 案例 | 原始工作 | 补充分析 | 价值 |
|------|---------|---------|------|
| Transformer 注意力分析 | "Attention Is All You Need" | 后续大量工作分析注意力头的作用、冗余性、不同层的行为差异 | 加深了对 Transformer 工作机制的理解 |
| BatchNorm 的真正原因 | BatchNorm 原论文声称减少了 Internal Covariate Shift | 后续工作证明效果来自优化landscape的平滑化 | 纠正了错误理解，推动了归一化方法发展 |
| Dropout 分析 | Dropout 原论文 | 后续工作分析其与贝叶斯推断的联系 | 建立了新的理论理解 |
| ResNet 为什么有效 | "Deep Residual Learning" | 后续从梯度流、集成学习等多个角度分析 | 深入理解了残差连接的本质 |

### 推荐方向举例

1. **轻量级 Vision Transformer 的行为分析**：目前大量工作在压缩 ViT，但很少有人系统分析剪枝/量化后模型的注意力模式如何变化、在什么样的输入上退化最严重。
2. **Prompt Engineering 的机制解释**：NLP 领域 Prompt 方法层出不穷，但对"为什么某个 prompt 格式更有效"的分析不够。可以设计系统实验探究 prompt 结构与模型内部表征的关系。
3. **数据增强方法的对比基准**：目前各种增强方法（Mixup/CutMix/RandAugment 等）在各自论文中评估标准不统一。统一基准下的系统性对比 + 深度分析是有价值的贡献。

### 风险评估

| 风险 | 级别 | 说明 | 应对 |
|------|------|------|------|
| 原创性质疑 | **高** | 审稿人可能认为"只是跑了更多实验" | 确保分析有新的 insight，不只是数据堆砌；提出新的理解角度或理论解释 |
| 被原作者抢先 | 中 | 原作者可能在后续版本中补充分析 | 选择发表一段时间、且作者似乎已经转向其他方向的论文 |
| 贡献不够清晰 | 中 | "更全面的分析"不如"新方法"容易被认可 | 提炼出明确的、反直觉的发现作为核心贡献 |

### 时间预估

| 阶段 | 时间 | 工作内容 |
|------|------|---------|
| 文献调研 | 2-3 周 | 精读目标论文 + 相关工作，找到分析缺口 |
| 复现基线 | 2-3 周 | 复现原始方法，确保能复现论文报告的结果 |
| 实验设计 | 1-2 周 | 设计新的评估维度和实验设置 |
| 实验执行 | 3-4 周 | 跑实验、收集数据、分析结果 |
| 论文写作 | 2-3 周 | 组织发现，形成有说服力的叙事 |
| **总计** | **10-15 周（约 3-4 个月）** | |

### 关键动作清单

1. **全面复现**：不是只跑通代码，而是确保理解每一步的含义
2. **发现新评估维度**：原论文没测试的场景（不同数据分布、不同模型规模、不同超参数范围）
3. **定位关键弱点**：找到原方法最脆弱的地方，分析原因
4. **提出新理解**：不是"这个方法在 X 上不好"，而是"这个方法在 X 上不好是因为 Y，这提示我们 Z"

## 策略 B：做一个小而硬的机制改进

### 定义

找到一个已有方法中明确的、具体的瓶颈点，提出一个简洁的、有理论动机的改进方案。关键在于"小"——改的是一个小组件或一个小环节，但"硬"——改进有明确的理由，实验能清晰证明有效性。

**核心思想**：不是发明新方法，而是在已有方法的某个关节上做一个精准的外科手术。

### 适用场景

- 你复现某篇论文时发现了明确的性能瓶颈或设计缺陷
- 你理解了某个模块的设计意图，但发现它可以被更好的方案替代
- 你有具体的、可验证的改进假设（不是"也许换个激活函数会更好"这种模糊猜测）

### 经典案例

| 案例 | 原始方法 | 改进点 | 为什么"小而硬" |
|------|---------|--------|--------------|
| Cosine Annealing LR | 标准的 step decay 学习率 | 改变学习率调度策略 | 只改了一行代码（调度器），但有明确的优化理论动机 |
| Label Smoothing | 标准 one-hot 标签 | 训练标签加噪声 | 改动极小，但有清晰的正则化理论支撑 |
| GELU 替代 ReLU | ReLU 激活函数 | 新激活函数 | 替换一个组件，但有概率论动机 |
| LayerScale | 标准 Transformer 残差连接 | 给残差加可学习缩放 | 一个极简改动，解决了深 Transformer 训练不稳定问题 |
| PatchMerger | ViT 的 patch 序列 | 合并冗余 patch 以降低计算量 | 明确的效率瓶颈 + 简洁方案 |

### 推荐方向举例

1. **轻量级注意力机制改进**：针对移动端 ViT 的注意力模块，当前方法在计算效率和表征能力之间有明确折中。提出一个更高效的注意力计算方式（比如基于哈希的稀疏注意力变体），在小模型上验证。
2. **训练稳定性改进**：许多轻量级模型训练不稳定，需要大量 trick。如果能找到一个简单的、有理论支撑的改进（比如改进初始化方案或归一化策略），使训练更鲁棒，这是一个好的"小而硬"贡献。
3. **少样本学习中的样本选择策略**：在 few-shot 场景中，哪些样本作为 support set 对结果影响巨大。提出一个更好的样本选择/加权策略，改进现有方法。

### 风险评估

| 风险 | 级别 | 说明 | 应对 |
|------|------|------|------|
| 改进幅度不够 | **高** | 审稿人可能认为改进太小 | 确保改进在统计上显著；强调改进的理论意义而非只看数值提升；在多个数据集/任务上验证 |
| 适用范围窄 | 中 | 改进可能只在特定设置下有效 | 在不同模型规模、数据集、任务上验证泛化性 |
| 已被他人发现 | 中 | 类似改进可能已经存在 | 做好文献调研，确保改进是新的或角度不同 |

### 时间预估

| 阶段 | 时间 | 工作内容 |
|------|------|---------|
| 文献调研 + 瓶颈定位 | 3-4 周 | 深入理解目标方法，精确定位瓶颈 |
| 改进方案设计 | 2-3 周 | 提出改进方案，理论分析可行性 |
| 代码实现 + 初步验证 | 2-3 周 | 实现改进，在小数据集上快速验证 |
| 全面实验 | 4-5 周 | 主实验 + 充分的消融实验 + 对比实验 |
| 论文写作 | 2-3 周 | 清晰呈现改进的动机、方法、分析 |
| **总计** | **13-18 周（约 4-6 个月）** | |

### 关键动作清单

1. **精确定位瓶颈**：不是"整体感觉不够好"，而是"在第 X 步，因为 Y 原因，导致 Z 问题"
2. **简洁方案**：改进方案应该可以用 3-5 行伪代码描述清楚。如果改动需要重写大量代码，说明改动不够"小"
3. **充分消融**：每个设计选择都要有对应的消融实验。"为什么是 A 而不是 B？"— 审稿人会问的问题，你要先回答
4. **理论动机**：即使不做严格的理论证明，改进方案也要有清晰的直觉/理论支撑。纯 empirical 的改进很难被认可

## 策略 C：组织已有思想成更完整的框架

### 定义

找到一个领域内存在多个独立的、碎片化的方法或思想，将它们统一在一个更抽象的框架下。不是发明全新的东西，而是发现已有方法之间的深层联系，提出一个整合性的视角。

**核心思想**：别人各自发明了锤子、螺丝刀、扳手，你来定义"工具箱"——统一接口、解释各自适用场景、发现缺失的工具。

### 适用场景

- 你发现某个领域内存在多个方法在做类似的事，但用了不同的术语和形式
- 你在不同子领域看到了相似的设计模式，但没人指出它们的共性
- 你能找到一个更高层次的抽象，把多个方法统一描述

### 经典案例

| 案例 | 整合前的碎片 | 统一框架 | 影响 |
|------|-------------|---------|------|
| 统一注意力视角 | 各种 Attention 变体（稀疏/局部/全局）| 统一的注意力计算框架，展示它们都是某种模式匹配 | 帮助理解注意力的本质，推动新变体设计 |
| 对比学习统一框架 | SimCLR/MoCo/BYOL/SimSiam 各自的损失函数 | 统一为"对齐 + 均匀"两个目标 | 一次性理解了所有对比学习方法的设计逻辑 |
| 正则化统一视角 | Dropout/BatchNorm/Weight Decay/Data Aug | 统一为对后验分布的不同近似 | 深刻理解了正则化的本质 |
| 脉冲神经网络与 ANN 统一 | SNN 和 ANN 被视为不同范式 | 证明 ANN 是 SNN 的特例，建立转换理论 | 打通了两个领域 |

### 推荐方向举例

1. **高效推理方法统一视角**：目前有知识蒸馏、剪枝、量化、动态推理等多种压缩/加速方法，它们各自独立发展。如果能找到一个统一的视角（比如都对应某种信息瓶颈优化），这是一个有深度的贡献。
2. **Prompt 方法分类学**：Hard prompt / soft prompt / prefix tuning / adapter 等方法看似不同，但可能共享某种深层结构。建立统一的分类框架，分析各自的理论动机和适用场景。
3. **数据增强的理论统一**：Mixup/CutMix/RandAugment/CMixup 等方法从不同角度增强数据，但缺乏统一的理论框架解释它们各自为什么有效以及在什么条件下失效。

### 风险评估

| 风险 | 级别 | 说明 | 应对 |
|------|------|------|------|
| 创新性依赖组合巧妙程度 | **高** | 如果框架太泛（万物归一但什么都没说）则无价值 | 框架必须有预测能力——能预测新方法或指出缺失的方法 |
| 工作量巨大 | **高** | 需要广泛阅读、深刻理解多个子领域 | 利用 AI 加速文献综述；选择范围适中的主题 |
| 实验难做 | 中 | 统一框架的价值体现在洞察而非实验 | 但仍需要精心设计的实验来验证框架的预测能力 |
| 对品味要求高 | **高** | 需要研究者有"看到本质"的能力 | 多读综述论文、多思考"这些方法的共同点是什么" |

### 时间预估

| 阶段 | 时间 | 工作内容 |
|------|------|---------|
| 广泛文献调研 | 4-5 周 | 覆盖目标领域的多个子方向，至少 30 篇论文 |
| 框架设计 | 3-4 周 | 提出统一抽象，验证能否覆盖已有方法 |
| 特例验证 | 2-3 周 | 用已有方法作为特例验证框架的完备性 |
| 预测验证 | 3-4 周 | 基于框架做出预测，设计实验验证 |
| 论文写作 | 3-4 周 | 清晰呈现统一框架 + 各特例 + 预测验证 |
| **总计** | **15-20 周（约 5-7 个月）** | |

### 关键动作清单

1. **找到共同抽象**：多个方法共享的数学形式或设计模式
2. **统一框架**：在这个抽象下，每个已有方法是框架的一个特例或配置
3. **特例证明**：严格展示已有方法如何从框架推导出来
4. **预测能力**：框架必须能预测新的、尚未被尝试的方法——然后用实验验证预测

## 策略选择决策树

根据以下条件选择最适合的策略：

```
开始
  │
  ├─ 你是否发现了一个明确的、具体的瓶颈？
  │    ├─ 是 → 你是否有具体的改进假设？
  │    │    ├─ 是 → 策略 B（小而硬的机制改进）
  │    │    └─ 否 → 继续向下
  │    └─ 否 → 继续向下
  │
  ├─ 你是否发现某个已有方法的分析明显不够？
  │    ├─ 是 → 你是否有新的分析角度？
  │    │    ├─ 是 → 策略 A（补上没做透的分析）
  │    │    └─ 否 → 继续向下
  │    └─ 否 → 继续向下
  │
  ├─ 你是否发现多个方法在做类似的事但没人统一过？
  │    ├─ 是 → 你是否能看到它们的共同抽象？
  │    │    ├─ 是 → 策略 C（组织成更完整的框架）
  │    │    └─ 否 → 多读论文，积累更多例子后再判断
  │    └─ 否 → 继续向下
  │
  ├─ 你的算力是否非常有限（无法训练大模型）？
  │    ├─ 是 → 优先策略 A（分析类工作算力需求最低）
  │    └─ 否 → 策略 B 或 C 均可
  │
  ├─ 你的时间是否紧张（需要快速产出）？
  │    ├─ 是 → 优先策略 A（周期最短，3-4 个月）
  │    └─ 否 → 可以尝试策略 B 或 C
  │
  └─ 首次做研究？
       ├─ 是 → 强烈推荐从策略 A 开始（风险最低、学习曲线最平缓）
       └─ 否 → 根据具体研究问题选择
```

### 简化版选择规则

| 你的情况 | 推荐策略 | 原因 |
|---------|---------|------|
| 首次做科研 | **策略 A** | 风险最低、周期最短、学习科研流程的最佳方式 |
| 算力非常有限 | **策略 A** | 分析类工作不需要大量训练，复现即可 |
| 发现了明确的瓶颈 | **策略 B** | 有具体目标，容易量化进展 |
| 文献积累丰富、有洞察力 | **策略 C** | 充分发挥知识管理优势 |
| 时间充裕（6个月+）| **策略 B 或 C** | 有足够时间做深入工作 |
| 需要快速出成果 | **策略 A** | 3-4 个月可完成 |

## 策略组合建议

三种策略并非互斥，可以灵活组合以增强论文贡献。

### 推荐组合模式

#### 组合 1：A + B（分析驱动的改进）

**最适合首次做科研的同学。**

1. 先用策略 A 深入分析某个已有方法，找到它的瓶颈和弱点
2. 基于分析发现，用策略 B 提出针对性的改进
3. 论文叙事：**"我们通过深入分析发现 X 存在 Y 问题，原因是 Z，因此提出改进方案 W"**

**优势**：分析部分提供了充分的动机，改进部分提供了具体的贡献。两者互补，论文说服力强。

**案例**：分析发现某 ViT 变体在小数据集上过拟合严重 → 定位原因是某个注意力头的冗余 → 提出自适应注意力剪枝方案。

#### 组合 2：A + C（分析驱动的统一）

**适合文献积累较多的同学。**

1. 先用策略 A 系统分析多个方法
2. 在分析过程中发现它们的共同模式
3. 用策略 C 提出统一框架
4. 论文叙事：**"通过对 X/Y/Z 方法的系统分析，我们发现它们的共同模式是 W，据此提出统一框架"**

**优势**：统一框架有扎实的实证基础，不是空泛的抽象。

#### 组合 3：B 单点突破

**适合有明确技术洞察的同学。**

1. 在复现或使用某方法时，发现一个明确的改进点
2. 深入理解改进的理论动机
3. 做充分的消融实验
4. 论文叙事：**"现有方法在 X 方面有明确局限，我们提出 Y 改进，理论动机是 Z，实验证明有效"**

**注意**：纯策略 B 需要改进效果足够显著，否则难以单独支撑一篇论文。建议至少辅以部分分析（策略 A 的元素）。

### 不推荐的组合

- **纯策略 C（首次科研）**：对研究品味要求太高，首次做科研很难驾驭
- **同时开展多个策略**：精力分散，建议先集中一个策略做好

## 实施节奏建议

| 时间 | 首次科研 | 有经验者 |
|------|---------|---------|
| 第 1-2 周 | 多方向快速调研，找策略 A 的候选目标 | 锁定方向，深入文献 |
| 第 3-4 周 | 精读 5-10 篇核心论文，确定目标 | 精读 10-15 篇，定位研究问题 |
| 第 5-8 周 | 复现 + 分析实验 | 设计方案，初步实现 |
| 第 9-12 周 | 深入分析，寻找 insight | 全面实验 + 消融 |
| 第 13-15 周 | 组织发现，开始写作 | 补充实验，开始写作 |
| 第 16 周+ | 完成初稿，迭代修改 | 完成初稿，迭代修改 |
