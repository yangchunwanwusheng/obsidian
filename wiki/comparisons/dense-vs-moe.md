# Dense 模型 vs MoE（混合专家）架构对比

> MoE 通过稀疏激活机制，在不增加实际计算量的情况下大幅扩展模型总参数容量，实现"以小模型的算力驱动大模型的能力"。

## 概述

传统 Dense Transformer 的每一层前馈网络（FFN）在处理每个 token 时会激活全部参数。Mixture of Experts（MoE）的核心理念是**稀疏激活**——维护多组并行的 FFN（称为"专家"），每个 token 只由其中少数专家处理，其余专家不参与计算。

这种设计的直觉来自现实世界：一家大型医院有 8 位不同科室的专家，但每位病人不需要让所有专家都看诊，只需根据症状路由到最相关的 1-2 位专家即可。MoE 将同样的逻辑应用到神经网络中。

关键区别在于：
- **Dense 模型**追求参数效率，每个参数在每个 token 上都被使用。
- **MoE 模型**追求容量效率，总参数量巨大但每个 token 只激活一小部分。

## 核心对比表

| 对比维度 | Dense Transformer | MoE（如 Mixtral 8x7B） |
|----------|-------------------|------------------------|
| **激活方式** | 全部参数参与每个 token 的计算 | 稀疏激活，每个 token 只用 top-k 个专家 |
| **总参数量** | 等于激活参数量（如 70B） | 远大于激活参数量（如 46.7B 总参数，含 8 个 7B 专家） |
| **激活参数/token** | 70B | ~12B（top-2 of 8） |
| **实际计算量** | 与总参数量成正比 | 接近小模型水平（仅激活参数参与计算） |
| **推理成本** | 高（全部参数计算） | 低（稀疏激活，计算量接近 1/6） |
| **训练难度** | 相对成熟稳定 | 路由不稳定、负载均衡、通信开销大 |
| **显存需求** | 所有参数需加载 | 所有专家权重需加载（即使不全部计算） |
| **代表模型** | GPT-3, LLaMA-2 70B, BERT | Mixtral 8x7B, GPT-4（推测）, Switch Transformer |
| **训练数据效率** | 一般 | 更高（更大容量吸收更多知识） |
| **部署灵活性** | 单机可部署 | 通常需要多卡/多机部署（参数量大） |

## MoE 的核心机制

### 路由（Router）

MoE 的关键组件是路由器——一个轻量级 MLP，为每个 token 计算分配到各专家的概率分布，然后选择概率最高的 top-k 个专家。

```
输入 token x
    ↓
Router（小型 MLP）
    ↓
输出概率分布: [Expert1: 0.05, Expert2: 0.42, Expert3: 0.38, ..., Expert8: 0.02]
    ↓
选择 top-2: Expert2 + Expert3
    ↓
加权求和: 0.42 × Expert2(x) + 0.38 × Expert3(x)
```

### 负载均衡问题

MoE 训练中最大的挑战是**路由崩塌**——所有 token 都被分配到同一两个专家，其余专家沦为"死专家"。解决方案包括：

- **辅助损失（Auxiliary Loss）**：在训练目标中加入负载均衡正则项，惩罚专家间负载不均
- **噪声注入**：路由决策前加入随机噪声，鼓励探索不同专家
- **专家容量限制**：强制每个专家处理的 token 数不超过上限

## MoE 的适用场景

### 最佳场景

| 场景 | 优势说明 |
|------|----------|
| **大规模预训练** | 用同等算力训练出参数量 6-8 倍的模型，吸收更多知识 |
| **推理吞吐优先** | 单 token 计算量小，批量推理吞吐量高 |
| **多任务混合部署** | 不同专家可能专精不同任务，天然支持多能力 |
| **算力有限但追求大模型能力** | 以小模型的 GPU 时间获得接近大模型的效果 |

### 不适用场景

| 场景 | 原因 |
|------|------|
| **单卡 / 资源受限部署** | 所有专家权重需加载到显存，实际显存需求远大于激活参数量 |
| **极低延迟推理** | 路由计算本身有额外开销，单 token 延迟可能不降反升 |
| **小规模模型（< 1B）** | 专家数量少时 MoE 的优势不明显，路由开销占比大 |
| **简单任务** | Dense 小模型已足够，MoE 的额外复杂度不值得 |

### 实际效果对比

以 Mixtral 8x7B 为例：
- 总参数约 46.7B，但每个 token 仅激活约 12B 参数
- 推理速度接近 12B Dense 模型
- 性能匹配或超过 LLaMA-2 70B（Dense 模型）
- 即用约 1/6 的计算量达到 6 倍参数量 Dense 模型的水平

## 选型建议

1. **训练预算有限但追求高性能** → MoE（以同等算力获得更大模型容量）
2. **部署环境显存充裕，追求推理吞吐** → MoE（稀疏激活降低单 token 计算量）
3. **部署环境显存紧张** → Dense 小模型（MoE 的总参数显存开销可能超标）
4. **追求极简架构和训练稳定性** → Dense（MoE 的路由和负载均衡增加工程复杂度）
5. **超大规模模型（> 100B）** → 几乎必然选择 MoE 或类似稀疏架构
