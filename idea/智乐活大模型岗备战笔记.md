## 第一部分：大模型基础 —— 一轮笔试 + 二轮必问

### 1.1 Transformer 核心原理

> ⭐⭐⭐⭐⭐ 这几乎是任何大模型岗的必问题，也是智乐活创始人（北大计算机博士）大概率会深挖的方向。

#### 自注意力机制（Self-Attention）

**问题通常是**："讲一下 Self-Attention 的计算过程"

**标准回答框架**：

```
输入：一个序列 X = [x1, x2, ..., xn]，每个 xi 是一个 d_model 维向量。

对于每个位置，计算三个向量：
- Query (Q) = X × W_Q    ← "我在找什么"
- Key   (K) = X × W_K    ← "我有什么特征"
- Value (V) = X × W_V    ← "我的实际内容"

计算流程（以第 i 个位置的注意力为例）：

第1步 — 计算注意力分数（点积）：
  score_i = Q_i · K_j^T   对所有 j = 1..n
  这衡量了"位置 j 对位置 i 有多重要"

第2步 — 缩放（除以 √d_k）：
  scaled_score = score_i / √d_k
  为什么要除？因为 d_k 很大时点积值会很大，
  经过 softmax 后梯度会变得很小（梯度消失），除以 √d_k 让方差稳定在1

第3步 — Softmax 归一化：
  attention_weights = softmax(scaled_score)
  得到每个位置的权重，加起来等于1

第4步 — 加权求和：
  output_i = Σ attention_weights_j × V_j
  把所有位置的 Value 按注意力权重加权平均

最终输出：每个位置都得到一个融合了全局上下文信息的表示。

复杂度：O(n² · d) —— 序列长度 n 的平方，这是长文本的瓶颈
```

#### 多头注意力（Multi-Head Attention）

```
为什么需要多头？
→ 单头只能学一种"关注模式"，多头可以同时关注不同方面。
   例如：一个头关注"语法结构"，另一个头关注"语义关系"，
   还有一个头关注"位置邻近关系"。

做法：并行计算 h 个独立的 Self-Attention（每个的 W_Q, W_K, W_V 不同），
      然后把结果拼接起来，再过一层线性变换。

MultiHead(Q,K,V) = Concat(head_1, ..., head_h) × W_O
其中 head_i = Attention(Q·W_Qi, K·W_Ki, V·W_Vi)
```

#### 为什么用 LayerNorm 而不是 BatchNorm？

```
这是经典面试题。

BatchNorm：对一个 batch 内所有样本的同一特征做归一化
  - 问题：序列长度不一时，batch 内的统计不稳定
  - 问题：推理时 batch=1 无法做 BN

LayerNorm：对一个样本内部所有特征做归一化
  - 不依赖 batch 大小 → 适合变长序列 → 适合 NLP
  - 训练和推理行为一致

记忆点：BN 是"横向归一化"（跨样本），LN 是"纵向归一化"（样本内部）
```

### 1.2 主流大模型架构对比

> 面试中常被问"你知道哪些主流模型？它们有什么区别？"

| 模型 | 架构类型 | 核心特点 | 适用场景 |
|------|---------|---------|---------|
| **GPT 系列** | Decoder-only | 自回归生成，单向注意力（只看左边） | 文本生成、对话、创作 |
| **BERT** | Encoder-only | 双向注意力，Masked LM 预训练 | 文本理解、分类、NER |
| **T5** | Encoder-Decoder | 统一的 text-to-text 框架 | 翻译、摘要、QA |
| **LLaMA** | Decoder-only | 开源，RoPE位置编码，SwiGLU激活 | 通用，社区生态好 |
| **ChatGLM** | Prefix Decoder | 双向+单向混合，中文友好 | 中文对话 |
| **DeepSeek** | MoE (混合专家) | 稀疏激活，效率高，MLA注意力 | 高效推理、长文本 |
| **Qwen** | Decoder-only | 阿里开源，中文能力强 | 中文场景、多模态 |

**口述时不要背书式罗列，要带上自己的理解**：
```
"我主要用过 GPT 和 DeepSeek。GPT 是 Decoder-only 架构，
适合生成类任务。DeepSeek 最近用 MoE —— 混合专家 —— 
每次推理只激活部分参数，效率和效果平衡得很好。
在适趣AI 的场景中，如果需要低成本、低延迟生成儿童阅读内容，
MoE 架构的 DeepSeek 可能比稠密模型更合适。"
```

### 1.3 大模型训练流程

> "从头到尾，一个大模型是怎么训练出来的？"

```
完整的训练管线：

1. 预训练 (Pre-training)
   目标：让模型学会语言的基本规律
   数据：海量文本（几TB到几十TB）
   任务：Next Token Prediction（预测下一个词）
   产出：Base Model（基座模型）
   例子：LLaMA、GPT-3 的基座版本

2. 监督微调 (SFT — Supervised Fine-Tuning)
   目标：让模型学会"对话"和"遵循指令"
   数据：高质量指令-回复对（几万到几十万条）
   方法：在预训练模型上继续训练
   产出：Chat Model
   例子：GPT-3.5-Turbo、LLaMA-2-Chat

3. 奖励建模 (Reward Modeling)
   目标：训练一个能评价"回复好不好"的模型
   数据：人工标注的偏好对（回复A > 回复B）
   方法：训练一个打分模型

4. 强化学习 (RLHF — RL from Human Feedback)
   目标：让模型的回复更符合人类偏好
   方法：用 PPO（近端策略优化）算法，
         最大化奖励模型的打分
   效果：更安全、更有用、更符合人类价值观

5. 可选：DPO (Direct Preference Optimization)
   简化版 RLHF，直接从偏好数据优化，
   不需要单独训练奖励模型，更简单稳定。
```

**面试中如果被问"RLHF 的原理"**：
```
"RLHF 的核心是让人工反馈来指导模型优化。
先收集人类对模型回复的偏好数据，训练一个奖励模型来
模拟人类的偏好判断。然后用强化学习，让模型调整参数
来最大化奖励模型的打分。

这和 SFT 的关键区别是：SFT 只告诉模型'标准答案是什么'，
RLHF 告诉模型'哪个更好'——它让模型学会了相对比较，
这对主观性强的任务（比如写得好不好、安不安全）特别重要。"
```

## 第二部分：大模型应用技术 —— 岗位核心

### 2.1 RAG（检索增强生成）⭐⭐⭐⭐⭐

> 这是"大模型技术应用"岗最高频且最深度的考点。
> 适趣AI 如果要确保内容的准确性（儿童教育不能出错），RAG 几乎是必用技术。

#### 完整 RAG 流水线

```
标准 RAG 流程（7个步骤）：

【离线阶段 — 知识库构建】
1. 文档加载 (Document Loading)
   将公司的知识库（教材、分级读物、生字表等）加载进来
   工具：LangChain 的 DocumentLoader、LlamaIndex 的 SimpleDirectoryReader

2. 文本切分 (Text Splitting)
   将长文档切成小块（chunk）
   关键参数：chunk_size（每块多大）、chunk_overlap（块之间重叠多少）
   为什么需要重叠？避免一个完整语义被切断在两个块里
   适趣AI 场景：可能按"每个生字 + 例句"来切分

3. 向量化 (Embedding)
   将每个 chunk 转成向量表示
   常用模型：text-embedding-3-small（OpenAI）、bge-large-zh（中文）
   产出：向量数据库中的索引

4. 存入向量数据库 (Vector Store)
   FAISS（轻量开源）、Milvus（生产级）、Chroma（易上手）

【在线阶段 — 查询处理】
5. 用户提问 → 同样过 embedding 模型变成向量

6. 相似度检索 (Retrieval)
   在向量数据库中找最相似的 top-k 个 chunk
   相似度度量：余弦相似度、欧氏距离
   进阶：MMR（最大边际相关性）去重，避免检索结果太相似

7. 增强生成 (Augmented Generation)
   把检索到的上下文 + 用户问题一起送给 LLM
   Prompt 模板：
   "根据以下参考资料回答用户问题。如果参考资料中没有相关信息，请如实告知。
    参考资料：
    {retrieved_chunks}
    用户问题：{user_question}"

   关键：LLM 的回复被"锚定"在检索到的真实知识上 → 减少幻觉
```

#### RAG 进阶：高级检索策略

```
基础 RAG 的问题：
- 关键词不匹配 → 向量检索也找不到相关内容
- top-k 固定 → 有些问题需要 3 个文档，有些需要 10 个
- 检索到的内容不相关 → LLM 被误导

改进方案（面试中可以提，体现深度）：
1. 混合检索 (Hybrid Search)
   向量检索 + 关键词检索（BM25）结合
   向量擅长语义，关键词擅长精确匹配

2. 查询重写 (Query Rewriting)
   用户问题先用 LLM 改写/扩展
   "这个字怎么读" → "某个汉字的标准读音和拼音是什么"

3. 重排序 (Re-ranking)
   粗检索 top-50 → 用更强的模型精排 top-5
   工具：Cohere Rerank、bge-reranker

4. Self-RAG / Corrective RAG
   检索后先评估相关性，不相关则重新检索或直接用 LLM 回答
```

#### RAG vs 微调 — 什么时候用哪个？

```
这是面试高频对比题。

RAG 的优势：
- 知识可以随时更新（改知识库就行，不用重新训练）
- 可解释性好（能看到检索到了什么文档）
- 幻觉更少（有明确的知识来源）
- 适合：知识密集型、需要引用来源、知识频繁更新的场景

微调的优势：
- 推理速度快（不需要检索步骤）
- 可以让模型学会特定的风格、格式、行为模式
- 适合：需要模型内化某种能力（比如"用小朋友的语气说话"）

适趣AI 场景：
RAG 用在哪？确保生字释义、拼音、笔顺等知识准确
微调用在哪？让模型学会"用3岁小孩能听懂的话解释"
两者不冲突，可以组合使用。
```

### 2.2 Prompt Engineering ⭐⭐⭐⭐

> "大模型技术应用"岗的日常工作之一。面试会考你实际的 prompt 设计能力。

#### 核心技巧

```
1. 结构化 Prompt
   ❌ "帮我写一篇儿童故事"
   ✅ 
   "你是一位儿童教育专家。请写一篇适合5岁孩子阅读的短故事。
    要求：
    - 全文使用100个以内的简单汉字
    - 每句话不超过10个字
    - 故事包含：开头、冲突、解决、结尾
    - 避免任何恐怖、暴力内容
    请以JSON格式输出，包含 title 和 content 字段。"

2. Few-Shot（给示例）
   在 prompt 中放 2-3 个示例，让 LLM 照猫画虎。
   关键是示例的质量和多样性，不是数量。

3. Chain-of-Thought (CoT)
   "请逐步思考，先分析问题，再给出答案。"
   让模型"show your work"，大幅提高复杂推理的准确率。

4. 角色设定
   "你是一位有10年经验的儿童阅读教育专家..."
   设定角色能让模型的行为更聚焦。

5. 输出格式控制
   用 JSON schema、正则约束或明确指令限制输出格式
   工具：guidance、outlines、instructor 库
```

#### 适趣AI 场景 Prompt 设计（面试可能直接让你现场设计）

**场景 1**：根据孩子的识字量生成一篇短文，只能使用孩子已经认识的字。

```
系统 Prompt 设计思路：
- 限制字表（"只能使用以下汉字：{生字列表}"）
- 控制难度（"每句话不超过X个字"）
- 质量检查（"生成后请自查：是否有不在字表中的字？"）
- 结构化输出（JSON，便于程序解析）

可能需要多轮生成 + 验证：
第一轮：生成文章
第二轮：LLM 自己检查是否用了未授权的字
第三轮：替换不合规的字
```

**场景 2**：给孩子读的内容做"适龄化改写"

```
系统 Prompt 设计思路：
- 保留核心意思，但用更简单的词
- 短句化：长句拆成多个短句
- 加拼音标注（可选）
- 维持趣味性（不能变成教科书式干巴巴的语言）
```

### 2.3 模型微调 ⭐⭐⭐

> 面经中高频出现的微调问题：LoRA 原理、QLoRA 区别、P-Tuning、全量微调对比。

#### LoRA（Low-Rank Adaptation）

```
核心思想：不直接修改原模型的权重矩阵 W，
而是加一个"旁路"：ΔW = A × B（A和B是两个低秩矩阵）

原来的计算：output = W × x
加LoRA后：  output = W × x + (α/r) × A × B × x
                         ↑ 原权重冻结  ↑ 只训练这个小矩阵

参数量对比（以 LLaMA-7B 为例）：
全量微调：70亿参数全部训练
LoRA(r=8)：只训练约 400万参数 = 原来的 0.06%

为什么有效？
核心假设：微调时的权重更新 ΔW 是低秩的
（不需要改变所有权重，只需要在低维子空间调整）

直观理解：原模型已经学会了语言，微调只是在"纠偏"，
         不需要大改，小修小补就行。
```

#### LoRA vs QLoRA vs P-Tuning

```
| 方法 | 核心思想 | 显存需求 | 效果 |
|------|---------|---------|------|
| 全量微调 | 训练所有参数 | 非常大 | 最好 |
| LoRA | 旁路低秩矩阵 | 中等 | 接近全量 |
| QLoRA | LoRA + 4bit量化 | 小 | 接近LoRA |
| P-Tuning v2 | 在每层加可训练prefix | 较小 | 好 |
| Adapter | 在Transformer层间加小网络 | 中等 | 好 |

实际选择：
- 有 A100 → LoRA，效果最好
- 只有 RTX 3090/4090 → QLoRA，性价比高
- 只调prompt → P-Tuning，最快但效果有限
```

### 2.4 AI Agent ⭐⭐⭐

> 适趣AI 如果有"个性化学习推荐"功能，大概率涉及 Agent 技术。

#### Agent 核心概念

```
Agent = LLM + 工具调用 + 记忆 + 规划

1. 工具调用 (Tool Use / Function Calling)
   LLM 不是直接回答，而是决定调用哪个外部工具
   例子：用户"帮我查一下这个字的笔顺"
         → LLM 决定调用 search_stroke_order(char)
         → 拿到结果后组织成自然语言回复

2. 记忆 (Memory)
   - 短期记忆：当前对话的上下文窗口
   - 长期记忆：向量数据库存储历史交互
   - 工作记忆：当前任务的中间状态

3. 规划 (Planning)
   ReAct 模式（Reasoning + Acting）：
   思考 → 行动 → 观察 → 思考 → 行动 → 观察 → ... → 完成
   
   例子（查天气 + 推荐衣服）：
   Thought: 我需要先查今天的天气
   Action: get_weather("北京")
   Observation: 晴天，15-25度
   Thought: 天气暖和，适合薄外套
   Action: recommend_clothes("春季", "晴天", "15-25")
   Observation: [推荐列表]
   Final Answer: 今天北京晴天，15-25度，建议穿薄外套...

4. 多Agent协作
   多个Agent分工协作，比如：
   - 内容生成Agent：写故事
   - 难度审核Agent：检查字词难度
   - 安全审核Agent：确保内容适合儿童
```

### 2.5 模型部署与推理优化

> 适趣AI 作为面向 C 端的产品，延迟和成本是关键。

```
推理加速技术（按难度排列）：
1. KV Cache — 缓存已计算的 Key/Value，避免重复计算
2. Flash Attention — 优化注意力计算的内存访问模式
3. 量化 (Quantization) — INT8/INT4 减少显存和计算量
4. vLLM — 高效的 LLM 推理引擎，PagedAttention
5. 投机采样 (Speculative Decoding) — 用小模型"猜"输出，大模型验证

成本优化：
- 缓存常见问题的回答（语义缓存）
- 分级模型：简单问题用小模型，复杂问题用大模型
- Batch处理：合并多个请求一起推理
```

**题1：实现一个简单的 RAG 查询函数**

```python
"""
题目：实现一个函数 rag_query(user_question, vector_store, llm_client)，
输入用户问题，返回基于 RAG 的增强回答。
"""
from typing import List

def rag_query(user_question: str, vector_store, llm_client) -> str:
    # 1. 将用户问题向量化
    query_embedding = get_embedding(user_question)  # 假设已有此函数
    
    # 2. 在向量数据库中检索 top-k 个最相似文档
    #    vector_store 可以是 FAISS/Milvus/Chroma 等
    retrieved_docs = vector_store.similarity_search(
        query_embedding, 
        k=5  # 取最相似的5个文档块
    )
    
    # 3. 拼接检索到的文档作为上下文
    context = "\n---\n".join([doc.content for doc in retrieved_docs])
    
    # 4. 构造增强 Prompt
    prompt = f"""你是一个教育助手。请根据以下参考资料回答用户问题。
如果参考资料中不包含相关信息，请如实说"我目前没有这方面的资料"。

参考资料：
{context}

用户问题：{user_question}

请用简洁清晰的中文回答："""
    
    # 5. 调用 LLM 生成回答
    response = llm_client.chat(prompt, temperature=0.3)
    
    return response
```

**题3：解释 Attention 中的 Q、K、V 分别代表什么**

```
Q (Query)：查询向量 —— "我在找什么"
  当前位置想知道"序列中哪些位置跟我相关"
  比如在"我爱吃苹果"中，"苹果"想知道前面哪个词跟它最相关

K (Key)：键向量 —— "我有什么特征可以匹配"
  每个位置对外提供的"检索标签"
  比如"苹果"的Key应该告诉我们它是"食物/水果"

V (Value)：值向量 —— "我的实际语义内容是什么"
  当注意力权重确定后，实际传递的信息
  V 才是最终要聚合的内容

类比（图书馆检索）：
- Q = 你的检索需求："我想找关于深度学习在 NLP 中应用的书"
- K = 每本书的索引标签："计算机/深度学习/NLP/机器学习"
- V = 书的具体内容

计算过程：用 Q 和每个 K 做匹配，得到每本书的相关性分数，
         用这些分数对 V（书的内容）加权求和。
```

### 4.2 场景设计题（很可能出现）

> 适趣AI 的面试官大概率会出一道和公司业务相关的场景题。

**示例场景 1**：
"我们要做一个功能：孩子对着 App 读一段文字，AI 判断他哪些字读对了、
哪些读错了、哪些读得不标准。你怎么设计这个系统？"

**回答框架**：
```
1. 语音识别 (ASR) 
   → 用 Whisper / 阿里云ASR / 讯飞 把孩子的朗读转成文字

2. 文本对齐
   → 将 ASR 输出和标准文本做对齐（Needleman-Wunsch 等序列对齐算法）
   → 找出替换/插入/删除的位置

3. 发音评估（可选，进阶）
   → 如果有原音频，可以用强制对齐 (Forced Alignment) 
   → 判断每个字的发音时长、音调是否标准
   → 工具：Kaldi / Montreal Forced Aligner

4. 反馈生成
   → 用 LLM 生成对孩子友好的反馈
   → "小明，你读得很棒！只有一个字需要注意：'蝴蝶'的'蝴'，
      跟'糊'不一样哦，来，跟老师再读一遍~"
   
5. 数据记录
   → 记录每个孩子的读音错误，形成个性化学习报告
```

**示例场景 2**：
"如果要用 AI 自动生成 100 篇适合 4 岁孩子的短故事，你怎么确保质量？"

**回答框架**：
```
1. Prompt 设计（前提约束）
   - 严格限定字表（只能用100个基础汉字）
   - 限定句式（每句不超过8个字）
   - 限定内容类型（动物、日常、自然 → 不要抽象概念）
   
2. 生成策略
   - 先用 LLM 批量生成主题列表
   - 每个主题生成 3-5 个故事变体
   - 加入随机种子确保多样性

3. 质量审核（多层过滤）
   ⭐ 这是关键 —— "怎么确保质量" 才是面试官想听的核心
   
   第1层：规则过滤
   - 检查是否有不在字表中的字
   - 检查句子长度、总字数
   - 检查是否有禁止词（暴力、恐怖等）
   
   第2层：LLM 自评
   - 用另一个 LLM（或同一个不同 Prompt）给每篇文章打分
   - 评分维度：趣味性、适龄性、教育价值、语言流畅度
   
   第3层：人工抽检
   - 随机抽取 10% 人工审核
   - 如果人工审核通过率 < 90%，回退优化 Prompt
```

## 第五部分：大模型岗速查表（面试前一天过一遍）

### 必背概念

```
□ Transformer: Q/K/V、Self-Attention、Multi-Head、LayerNorm、位置编码
□ 训练流程: Pre-training → SFT → RLHF → DPO
□ 微调方法: LoRA（低秩分解）、QLoRA（LoRA+量化）、P-Tuning（prefix tuning）
□ RAG: Document Loading → Chunking → Embedding → Vector Store → Retrieval → Generation
□ Agent: ReAct 模式 = 思考 → 行动 → 观察 → 循环
□ Prompt: Few-Shot、CoT、结构化输出、角色设定
□ 推理优化: KV Cache、Flash Attention、量化、vLLM
□ 幻觉: 成因（训练数据噪声、过度泛化、缺少约束）、解法（RAG、Prompt约束、后处理验证）
□ 评估: BLEU/ROUGE（词重叠）、Perplexity（语言模型困惑度）、RLHF reward（人类偏好）
□ MoE: 混合专家，每次推理只激活部分参数，DeepSeek/Mixtral 在用
```
