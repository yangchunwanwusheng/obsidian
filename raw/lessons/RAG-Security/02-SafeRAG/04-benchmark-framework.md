# 04 — 基准框架与评测指标

> 本节介绍 SafeRAG 的数据集构建流程、评测 Pipeline、14 个 RAG 组件矩阵以及攻击特定评测指标体系。

---

## SafeRAG 数据集构建流程

### 正文翻译

SafeRAG 基准数据集**以人工构造为主、LLM 辅助**，构建流程如下：

**第 1 步：基础数据集收集**
收集一批**中文新闻文章**作为基础数据集，构建全面的"问题-上下文"对。

**第 2 步：攻击目标文本选择**
从基础数据集中选择适合攻击构造的文本。

**第 3 步：攻击文本生成**
针对四种攻击类型分别生成攻击文本：

| 攻击类型 | 构造方法 |
|---------|---------|
| **银噪声** | 从正确证据中删除/修改部分关键信息，保留表面相关性 |
| **上下文间冲突** | 人工编写与正确证据相矛盾但表面合理的内容 |
| **软广告** | 在证据中嵌入权威来源伪装的推广信息 |
| **白 DoS** | 构造虚假安全警告文本 |

**第 4 步：质量验证**
人工检查每条攻击文本的有效性。

### 覆盖的评测任务

- **领域特定问答**：针对特定领域知识库的问答
- **新闻问答**：基于新闻文章的开放域问答

> [!note] 译者注（非原文）
> SafeRAG 的数据集构造策略与 PoisonedRAG 形成鲜明对比——SafeRAG 以人工构造为主（确保质量但扩展性弱），PoisonedRAG 以 LLM 自动生成为主（扩展性强但可能产生不自然文本）。这反映了"评测基准"与"攻击方法"的不同定位——评测基准必须确保攻击文本的有效性和自然性，而攻击方法追求效率和自动化。

## 评测 Pipeline

### 正文翻译

SafeRAG 模拟攻击场景的流程：

```
攻击文本注入 → 检索器 → 检索上下文 → 过滤器 → 过滤后上下文 → 生成器 → 评估
    ↑              ↑                     ↑                   ↑
 知识库注入    检索阶段注入            过滤阶段注入         全部阶段
```

攻击可在 **4 个阶段**注入，逐级递进：

| 注入阶段 | 英文缩写 | 说明 | 攻击难度 |
|---------|---------|------|---------|
| **知识库** | KB | 攻击文本混入知识库，需先被检索到 | 最高（需通过检索） |
| **检索上下文** | RC | 攻击文本直接插入 Top-k 检索结果 | 中等（已过检索关） |
| **过滤后上下文** | FC | 攻击文本绕过过滤器，直接出现在最终上下文 | 最低（直通生成器） |
| **全阶段** | All | 同时测试各阶段的端到端脆弱性 | — |

> [!tip] 理解提示（非原文）
> 多阶段注入设计是 SafeRAG 的另一个核心特色。它回答了一个关键问题：**攻击在哪个阶段注入最有效？** 答案直接影响防御策略——如果你知道越靠近生成器的攻击越危险，就应该将防御重心后移。

## 14 个代表性 RAG 组件

SafeRAG 在以下组件上系统性评估：

### 检索器（4 种）

| 检索器 | 类型 | 特点 |
|--------|------|------|
| **DPR** | 密集段落检索 | 纯语义匹配 |
| **BM25** | 稀疏检索/关键词匹配 | 纯词汇匹配 |
| **Hybrid** | DPR + BM25 混合 | 语义 + 词汇 |
| **Hybrid-Rerank** | 混合 + Cross-Encoder 重排序 | 检索后再精排 |

### 过滤器（2 种）

| 过滤器 | 类型 | 说明 |
|--------|------|------|
| **NLI** | 自然语言推理过滤器 | 判断上下文是否支持回答问题 |
| **SKR** | 压缩器 | 合并冗余信息 |

### 生成器（8 种）

GPT-3.5-Turbo、GPT-4、DeepSeek-V2、Baichuan-13B、ChatGLM3-6B、Qwen-7B、Qwen-14B、LLaMA-2-7B（中文微调版）

## 评测指标体系

### 正文翻译

SafeRAG 针对不同攻击类型设计了**专用指标**，并与人工判断进行了对齐验证：

| 指标 | 含义 | 适用攻击 | 期望方向 |
|------|------|---------|---------|
| **F1 (correct)** | 正确答案的 F1 得分 | 所有类型 | 越高越好 |
| **F1 (incorrect)** | 错误/无关答案的 F1 得分 | 所有类型 | — |
| **F1 (avg)** | (F1_correct + F1_incorrect) / 2，答案多样性 | **噪声** | 越高越好（多样性好） |
| **ASR** | 攻击成功率 | **冲突、毒性** | 越低越好 |
| **AFR** | 攻击失败率 | **DoS** | 越低越好 |
| **RA** | 检索准确率（Retrieval Accuracy） | 所有类型 | 越高越好 |

**评估器选择**：**DeepSeek 被选为统一评估器**，因为其评估结果与人工判断的一致性最高：

| 指标 | 人工一致性 |
|------|----------|
| F1 (correct) / Silver Noise | 89.97% |
| F1 (incorrect) / Silver Noise | 96.22% |
| ASR / Inter-Context Conflict | 95.65% |
| ASR / Soft Ad | **100%** |
| AFR / White DoS | **100%** |

> [!note] 译者注（非原文）
> 使用 LLM 作为评估器（LLM-as-a-Judge）是现代 NLP 评测的常见做法，但 SafeRAG 做得更严谨——它先验证了评估器与人工判断的一致性。DeepSeek 的一致性在 89-100% 之间，这对于自动评测来说是非常高的水平。尤其是软广告和白 DoS 的评估与人工达到了 100% 一致——说明这些攻击的效果非常明确、易于判断。

## 关键设计特点总结

| 特点 | 说明 |
|------|------|
| **人工为主** | 数据集以人工构造为主，确保攻击文本的自然性和有效性 |
| **多阶段注入** | 覆盖 KB/RC/FC/All 四个注入阶段 |
| **攻击特定指标** | 不同攻击类型使用不同指标，更精确地衡量危害 |
| **人工对齐验证** | 所有自动指标均与人工判断进行了对齐 |
| **14 组件矩阵** | 4 检索器 × 2 过滤器 × 8 生成器，覆盖全面 |
