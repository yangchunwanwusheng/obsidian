# LLM 多智能体综述中文精读（第 4 篇-03）：系统内通信

> 本篇覆盖 **Section 4 System Internal Communication**，把镜头从全局通信架构拉到单次通信行为的微观机制。作者用四个维度来分析系统内通信：策略（Strategies）、范式（Paradigms）、对象（Objects）和内容（Content）。

## 本章导读

> [!note] 译者注（非原文）
> 如果说 Section 3 是在画组织架构图和通信网络图，那么 Section 4 就是在分析单次"对话"本身是怎么发生的。四个维度的逻辑关系是：
>
> - **Strategies（策略）**：谁在什么时候发言？——控制发言的时序和调度；
> - **Paradigms（范式）**：信息以什么形式传递？——定义通信的基本模式；
> - **Objects（对象）**：对谁说话？——明确通信的目标受众；
> - **Content（内容）**：说了什么？——通信携带的信息类型。
>
> 这四个维度合在一起，可以完整描述"一次 agent 通信行为是如何发生的"。

## 正文翻译

> [!warning] 易混点（非原文）
> 从这里开始进入原文翻译。"系统内通信"（System Internal Communication）与"系统间通信"（System-Level Communication）的区别是本章理解的关键——前者关心"怎么说话"，后者关心"怎么组织说话"。

### 原文标题：4 System Internal Communication

当系统间通信的架构和目标确定后，接下来的问题是：在具体的通信行为中，agent 之间是如何互动的？作者将系统内通信的微观机制分解为四个维度：通信策略、通信范式、通信对象和通信内容。

#### 4.1 Communication Strategies（通信策略）

通信策略回答的问题是：**在多个 agent 之间，谁在什么时候发言？** 作者识别出三种核心策略：

##### 4.1.1 One-by-One（顺序发言）

**定义**：agent 按照预定义的顺序依次发言。每个时刻只有一个 agent 在"说话"，其他 agent 在"听"。

**工作机制**：
- 发言顺序可以是固定的（如 agent A -> agent B -> agent C -> agent A -> ...），也可以是动态决定的（如根据上一轮的发言内容决定下一个发言者）；
- 每个发言者可以访问之前所有轮次的历史信息；
- 通信的推进由发言顺序驱动，直到满足停止条件（如达到最大轮次、达成共识、任务完成）。

**典型实现**：
- **固定轮询（Round-Robin）**：agent 按固定顺序轮流发言，最简单也最常用；
- **基于角色的顺序**：不同角色的 agent 按照特定的发言序列进行（如"管理者提出任务 -> 执行者报告进展 -> 评审者给出反馈"）；
- **动态选择**：根据上下文动态决定下一个发言者（如通过投票、评分或一个专门的调度 agent 来选择）。

**优势**：
- 信息传递有序，每个 agent 都能看到完整的对话历史；
- 容易实现和调试——通信过程是可预测的、可追溯的；
- 避免了发言冲突和信息混乱。

**劣势**：
- 效率低下——agent 在等待发言时闲置；
- 不适合需要并行处理的场景；
- 顺序本身可能引入 bias——先发言的 agent 会对后发言的产生锚定效应（anchoring effect）；
- 当 agent 数量较多时，轮次延迟显著增加。

> [!tip] 理解提示（非原文）
> One-by-One 是最常见的策略，因为它最简单也最可控。几乎所有早期的 LLM-MAS 都采用了这种策略。当你的系统只有 2-5 个 agent 时，One-by-One 通常就够用了。

##### 4.1.2 Simultaneous-Talk（并行发言）

**定义**：多个 agent 可以同时"发言"，彼此之间不一定等待对方完成。

**工作机制**：
- 所有活跃的 agent 同时接收相同的输入（或各自相关的输入）；
- 每个 agent 独立生成响应，不等待其他 agent 完成；
- 所有 agent 的响应被收集后，作为下一轮的上下文输入。

**典型实现**：
- **全并行**：每一轮所有 agent 同时基于相同的输入生成响应，然后收集所有响应作为下一轮的输入；
- **条件并行**：只有满足特定条件的 agent 才发言（如"对当前话题有相关知识的 agent"、"被分配了当前子任务的 agent"）；
- **异步并行**：agent 独立运行，有新信息时随时发言，不限定时序。

**优势**：
- 效率高——agent 不需要等待，可以充分利用并行计算资源；
- 更接近真实的多方对话（如会议讨论、头脑风暴）；
- 每个 agent 的响应不受其他 agent 当前轮次的影响，减少了锚定效应。

**劣势**：
- 信息可能混乱——同时收到多个 agent 的发言，当前的 LLM 可能难以有效处理大量并发信息；
- 上下文窗口压力大——每轮所有 agent 的发言都需要塞进上下文，token 消耗快速增长；
- 可能出现冗余——多个 agent 可能给出相似的回应，"大家都在说同样的话"；
- 缺乏对话的连贯性——agent A 没有机会即时回应 agent B 的发言。

##### 4.1.3 Simultaneous-Talk-with-Summarizer（带汇总者的并行发言）

**定义**：多个 agent 并行发言后，由一个专门的 Summarizer agent 对所有发言进行汇总、去重和提炼，然后将精炼后的信息传递给下一轮。

**工作机制**：
- 第一步：多个 agent 并行生成各自的响应；
- 第二步：Summarizer agent 读取所有响应，生成一个结构化的摘要；
- 第三步：摘要成为下一轮所有 agent 的共享输入。

**典型实现**：
- **LLM 作为 Summarizer**：使用一个专门的 LLM 调用来完成汇总任务；
- **结构化汇总**：Summarizer 不仅生成自由文本摘要，还提取关键决策点、共识项、分歧项和待办事项；
- **分层汇总**：在大规模系统中，每个子群先有自己的 Summarizer，各组摘要再由上层 Summarizer 汇总。

**优势**：
- 结合了并行发言的效率和顺序发言的信息清晰度；
- Summarizer 可以过滤冗余和无关信息，降低后续轮次的上下文压力；
- 结构化摘要可以帮助 agent 更快地理解和响应。

**劣势**：
- Summarizer 本身是新的单点故障和性能瓶颈；
- 汇总过程可能丢失重要信息（summarizer 的偏见和遗漏）；
- 增加了系统复杂度和通信延迟（多了一步汇总操作）；
- Summarizer 的质量直接影响整个系统的通信质量。

##### 三种策略的对比与选型建议

作者通过一个对比表格总结了三种策略的核心取舍：

| 策略 | 效率 | 信息完整性 | 实现复杂度 | 适用场景 |
|------|------|-----------|-----------|---------|
| One-by-One | 低 | 高 | 低 | agent 数量少、需要有序对话 |
| Simultaneous-Talk | 高 | 中（可能混乱） | 中 | 需要并行加速、agent 数量多 |
| Simultaneous-Talk-with-Summarizer | 中 | 中（取决于 Summarizer 质量） | 高 | 大规模 agent 群、需要控制上下文窗口 |

> [!note] 译者注（非原文）
> 这三种策略代表了"有序 vs. 并行"这个经典张力在 LLM-MAS 通信中的具体体现。在实际系统中，选择哪种策略取决于你的瓶颈在哪里——如果是 agent 数量太多导致的延迟，选并行；如果是上下文窗口不足导致的截断，选带汇总者的并行；如果通信质量比效率更重要，选顺序。

#### 4.2 Communication Paradigms（通信范式）

通信范式回答的问题是：**信息以什么形式在 agent 之间传递？** 作者借鉴了传统 MAS 和分布式系统的经典概念，识别出三种核心范式：

##### 4.2.1 Message Passing（消息传递）

**定义**：agent 之间通过显式的、定向的消息进行点对点通信。消息有明确的发送者和接收者。

**工作机制**：
- agent A 生成一条消息，明确指定接收者为 agent B；
- 消息包含特定的内容（如请求、回复、通知）；
- 消息传递可以是同步的（发送者等待回复）或异步的（发送者继续执行）。

**在 LLM-MAS 中的体现**：
- agent 之间的直接对话：一个 agent "@另一个 agent" 发送特定问题或指令；
- API 风格的通信：agent 通过结构化接口调用其他 agent 的能力；
- 事件通知：agent 向指定订阅者发送状态更新或事件通知。

**优势**：
- 通信精准——信息只流向需要它的 agent；
- 隐私性好——消息只对发送者和接收者可见；
- 易于追踪和审计——每条消息的发送者和接收者是明确的。

**劣势**：
- 需要 agent 知道"该找谁"——要求 agent 具备对系统拓扑的认知；
- 一对一通信模式在信息需要广播时效率低；
- 如果接收者不可用或响应慢，发送者可能被阻塞。

##### 4.2.2 Speech Act（言语行为）

**定义**：借鉴 Austin 和 Searle 的言语行为理论（Speech Act Theory），将每一次通信视为一种具有特定意图的"行动"，而不仅仅是信息传递。

**核心概念**：言语行为理论区分了话语的三个层面：
- **言内行为（Locutionary Act）**：说话本身——"我说了什么"；
- **言外行为（Illocutionary Act）**：说话者的意图——"我通过说这话想做什么"（如断言、请求、承诺、命令、声明）；
- **言后行为（Perlocutionary Act）**：说话的效果——"我的话对听者产生了什么影响"。

**在 LLM-MAS 中的体现**：
- **请求（Request）**：一个 agent 请求另一个 agent 执行某个行动或提供某些信息；
- **承诺（Commitment）**：一个 agent 承诺完成某项任务或遵守某个约束；
- **断言（Assertion）**：一个 agent 声称某个命题为真；
- **声明（Declaration）**：一个 agent 宣布一项决定或状态变更；
- **问题（Question）**：一个 agent 向另一个 agent 提出疑问。

作者指出，LLM 天然具备理解和生成不同言语行为的能力——它们可以识别一个请求是"礼貌的请求"还是"强硬的命令"，也可以根据语境生成恰当的言语行为。

**优势**：
- 通信的语义层次更高——不只是"传了什么字"，而是"想表达什么意图"；
- 支持复杂的社会交互——协商、说服、争论等高级交互模式；
- 更自然地映射到人类通信方式。

**劣势**：
- 意图识别可能出错——LLM 可能误判一条消息的言语行为类型；
- 意图与效果之间的差距——说话者的意图不一定被接收者正确理解；
- 需要额外的机制来追踪承诺和声明的履行情况。

> [!tip] 理解提示（非原文）
> Speech Act 范式的一个关键价值在于：它把通信从"数据交换"提升到了"社会行动"的层次。当你用 Speech Act 框架分析 LLM-MAS 通信时，你会关注的不是"传了多少 token"，而是"谁会因为这条消息做出什么承诺、改变什么信念"。

##### 4.2.3 Blackboard（黑板）

**定义**：多个 agent 共享一个公共的"黑板"（共享工作空间）。agent 可以在黑板上写入信息、读取他人写入的信息、修改或批注已有信息。

**在 LLM-MAS 中的体现**：
- **共享记忆/上下文**：所有 agent 可以看到同一个对话历史或共享状态；
- **共享文档/代码**：多个 agent 协作编辑同一个文档或代码库；
- **共享知识库**：agent 向一个公共知识库添加、查询和更新信息；
- **结构化黑板**：黑板被组织为特定的结构（如任务列表、状态机、决策树），方便 agent 理解和操作。

**优势**：
- 解耦——agent 不需要知道彼此的存在，只需与黑板交互；
- 持久化——黑板上的信息在 agent 会话之间持续存在；
- 灵活——新的 agent 可以随时加入，只需学会读写黑板；
- 适合异步协作——agent 可以在不同时间与黑板交互。

**劣势**：
- 黑板上的信息可能变得混乱和无序，需要专门的维护机制；
- 黑板的规模和复杂度增长时，agent 难以高效找到相关信息；
- 读写冲突——多个 agent 同时修改同一区域的信息；
- 安全问题——所有 agent 都能看到黑板上的所有信息。

##### 三种范式的关系

作者强调，三种范式不是互斥的。在实际的 LLM-MAS 中：
- **Message Passing** 常用于紧急、私密的协调（如"帮我检查这段代码"）；
- **Speech Act** 常用于需要明确意图的交互（如"我承诺在周五前完成"）；
- **Blackboard** 常用于全局状态共享和长程协作（如"这是当前项目进度"）。

一个设计良好的 LLM-MAS 通常会组合使用多种范式。

#### 4.3 Communication Objects（通信对象）

通信对象回答的问题是：**agent 在与谁通信？** 它不仅包括"其他 agent"，还涵盖了更广泛的受众。

##### 4.3.1 Self（自我通信）

**定义**：agent 与自己进行内部对话，用于推理、反思和自我修正。

**在 LLM-MAS 中的体现**：
- **Chain-of-Thought（CoT）**：agent 在生成最终输出前进行内部推理；
- **Self-Reflection**：agent 反思自己的输出质量并自我修正；
- **Internal Monologue**：agent 在采取行动前进行内部独白，评估不同行动方案的优劣。

作者指出，虽然自我通信严格来说不是"多智能体通信"，但它对 agent 与其他 agent 通信的质量至关重要——一个能良好"自我通信"的 agent，才能向其他 agent 发送更清晰、更有价值的信息。

##### 4.3.2 Other Agents（其他 agent）

**定义**：agent 之间相互通信。这是多智能体通信最核心的形式。

**通信细分**：
- **一对一（1-to-1）**：两个 agent 之间直接对话；
- **一对多（1-to-N）**：一个 agent 向多个 agent 广播信息；
- **多对一（N-to-1）**：多个 agent 向同一个 agent 发送信息（如多个 worker 向 manager 汇报）；
- **多对多（N-to-M）**：任意 agent 之间的自由通信。

##### 4.3.3 Environment（环境）

**定义**：agent 通过改变环境状态来实现间接通信，而不是直接发送消息。

**在 LLM-MAS 中的体现**：
- **共享工具的输出**：一个 agent 调用工具产生的结果，可以被其他 agent 观察到；
- **代码/文档变更**：agent 提交的代码或文档变更，构成一种环境级的通信；
- **数据库更新**：agent 对共享数据库的修改，被其他 agent 读取。

这种通信模式的灵感来自自然界的**间接通信（Stigmergy）**——如蚂蚁通过信息素标记路径，后来的蚂蚁根据信息素的浓度选择路线。

> [!tip] 理解提示（非原文）
> 通过环境通信的一个重要优势是：它天然支持异步和解耦。一个 agent 对环境做出的修改，可以在之后的任意时间点被另一个 agent 发现和利用——不需要两者同时在线。

##### 4.3.4 Human（人类）

**定义**：agent 与人类用户之间的通信。这是 LLM-MAS 中一个日益重要的方向——Human-in-the-Loop。

**通信形式**：
- **信息请求**：agent 遇到不确定或超出能力范围的问题时，向人类用户寻求帮助；
- **进度报告**：agent 向用户汇报当前进展和关键决策；
- **确认请求**：agent 在执行高风险操作前请求人类确认；
- **教育性输出**：agent 向用户解释自己的推理过程和决策依据。

**设计原则**：
- **适时介入**：决定何时需要人类介入（频率太低可能错过关键决策，太高则失



去多智能体自动化的意义）；
- **信息密度**：向人类呈现的应该是经过提炼和组织的关键信息，而非原始的 agent 对话流；
- **可控性**：人类应该能够在不同粒度上控制和干预 agent 的行为。

#### 4.4 Communication Content（通信内容）

通信内容回答的问题是：**agent 通信传递的是什么类型的信息？** 作者将其分为两大类：

##### 4.4.1 Explicit Communication（显式通信）

**定义**：agent 通过语言或结构化格式明确表达的信息。

**在 LLM-MAS 中的具体形式**：

（1）**自然语言文本**
这是 LLM-MAS 中最主要也最自然的通信内容形式。agent 通过自然语言表达：
- 推理过程和中间结论；
- 对任务的分解和分配；
- 对其他 agent 的请求和反馈；
- 对当前状态的理解和判断。

（2）**结构化输出**
除自然语言外，agent 也可以输出结构化的信息：
- JSON / YAML 格式的数据；
- 表格和列表；
- 代码片段和伪代码；
- 数学公式和逻辑表达式。

（3）**引用和证据**
agent 可能引用外部来源来支持自己的论点：
- 论文引用；
- 网页链接；
- 数据来源说明；
- 工具调用的返回结果。

**显式通信的优势**：
- 信息明确，接受者可以直接理解和处理；
- 易于记录、审计和调试；
- 支持复杂的语义表达。

**显式通信的局限**：
- token 消耗大——自然语言通信的内容量可以快速增长；
- 可能包含噪声和冗余——不是所有 agent 说的话都有价值；
- 表达可能不精确——自然语言的模糊性可能导致误解。

##### 4.4.2 Implicit Communication（隐式通信）

**定义**：agent 不通过直接的语言表达，而是通过行为、状态变化或上下文暗示来传递信息。

**在 LLM-MAS 中的具体体现**：

（1）**行为信号**
- **沉默即态度**：一个 agent 选择在某些话题上沉默，本身就传递了信息（如默认同意、不相关、缺乏信息）；
- **工具选择**：agent 选择调用哪个工具、以什么参数调用，反映了它对任务的理解和策略；
- **响应延迟**：agent 的响应速度快慢可以反映任务难度或其对任务的把握程度。

（2）**中间推理的可观察性**
在 LLM-MAS 中，一个 agent 的推理过程（如 Chain-of-Thought）如果对其他 agent 可见，就构成隐式通信：
- 暴露了 agent 的思考路径和偏误；
- 让其他 agent 有机会发现推理中的逻辑漏洞；
- 减少了重新推理的冗余。

（3）**状态变化**
- agent 更新了共享记忆或知识库；
- agent 改变了自己的角色或行为模式；
- agent 加入或退出系统。

（4）**上下文暗示**
- 对话历史的长度和复杂度反映任务进展；
- 某类信息的反复出现暗示其重要性；
- agent 之间的沉默或回避某些话题反映潜在冲突。

**隐式通信的优势**：
- 不增加额外的通信负载（agent 不需要为通信专门生成消息）；
- 可以传递"言外之意"——那些不方便或不需要明确说出的信息；
- 更接近人类社交中的微妙信号。

**隐式通信的局限**：
- 容易被忽视或误读——不是所有 agent 都能正确解读隐式信号；
- 难以显式控制——隐式通信的效果依赖于接收者的解读能力；
- 不适合传递关键决策信息——重要信息应该显式确认。

> [!tip] 理解提示（非原文）
> 显式通信和隐式通信的关系，类似于人类交流中"说了什么"（verbal）和"怎么做、什么态度、什么表情"（non-verbal）的关系。在 LLM-MAS 中，研究者正在探索如何让 agent 既能发送清晰的语言消息，也能"读懂"其他 agent 的行为和状态中隐含的信息。

##### 两种通信内容的互补性

作者强调，显式通信和隐式通信不是二元对立的，而是互补的：

- 关键决策和承诺：应该用显式通信（避免误解）；
- 状态同步和氛围感知：可以用隐式通信（降低通信开销）；
- 设计良好的系统：会在两种通信模式之间找到合适的平衡。

## 本章小结

> [!note] 译者注（非原文）
> Section 4 提供了分析 LLM-MAS 微观通信行为的完整工具箱。四个维度的核心判断：
>
> 1. **策略（Strategies）**：One-by-One 安全但慢，Simultaneous 快但可能乱，加 Summarizer 是折中方案。选择取决于你的瓶颈是延迟、上下文窗口还是通信质量。
>
> 2. **范式（Paradigms）**：Message Passing 适合精准协调，Speech Act 适合意图明确的交互，Blackboard 适合全局共享。好的系统会组合使用。
>
> 3. **对象（Objects）**：通信不限于 agent 之间——自我通信、环境通信、人类通信都是系统内通信的重要组成部分。
>
> 4. **内容（Content）**：显式通信是主体，但隐式通信正在成为一个重要的研究方向——让 agent 能"读懂"行为背后的信息。

## 思考题

### 题目 1：什么场景下你会选择 Simultaneous-Talk 而不是 One-by-One？反过来呢？

> [!tip] 理解提示（非原文）
> 考虑两种极端场景：2 个 agent 进行数学证明辩论 vs. 20 个 agent 分别独立评估一篇文章的质量。

> [!note] 译者注（非原文）
> 选 Simultaneous-Talk 的场景：agent 数量多（5+）、任务可以并行分解（如独立评估）、通信效率比精确的对话顺序更重要、上下文窗口足以容纳所有并行发言。选 One-by-One 的场景：agent 数量少（2-3）、任务需要有序的对话和逐步推导（如辩论、多步推理）、每个 agent 的发言高度依赖于上一个发言者的内容。

### 题目 2：隐式通信在实际 LLM-MAS 中有什么具体用途？请给出一个例子。

> [!warning] 易混点（非原文）
> 隐式通信不是"agent 不说话了"，而是"agent 通过非语言的途径传递了信息"。

> [!note] 译者注（非原文）
> 一个典型例子：在一个代码审查多智能体系统中，如果 Security Reviewer agent 在审查某段代码后，选择不调用"报告漏洞"工具，而是直接进入下一段代码的审查——这个"没有报告漏洞"的行为本身就是一个隐式信号，暗示该段代码在安全维度上是 clean 的。其他 agent 可以观察到这个行为（通过共享状态或行为日志），而无需 Security Reviewer agent 显式说出"这段代码安全"。
