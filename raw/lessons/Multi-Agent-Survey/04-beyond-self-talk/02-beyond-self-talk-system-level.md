# LLM 多智能体综述中文精读（第 4 篇-02）：系统间通信

> 本篇覆盖 **Section 3 System-Level Communication**，是全文最核心的分析章节之一。作者从三个维度展开系统间通信的设计空间：架构（Architecture）、通信目标（Communication Goals）和通信协议（Communication Protocols）。

## 本章导读

> [!note] 译者注（非原文）
> Section 3 是这篇论文最有原创性的分析框架。作者不是简单地罗列现有的多智能体系统，而是先建立起"系统间通信"的分析维度，然后用这些维度去审视和归类现有工作。
>
> 三个维度的逻辑关系是：
> - **Architecture** 回答"怎么连"——agent 之间的拓扑结构是什么？
> - **Goals** 回答"为什么连"——agent 通信的目的是什么？
> - **Protocols** 回答"用什么连"——agent 通信的技术标准是什么？
>
> 这三个维度层层递进：先有架构（连接方式），再赋予目标（连接目的），最后用协议落地（连接技术）。

## 正文翻译

> [!warning] 易混点（非原文）
> 从这里开始进入原文翻译。本节的术语比较密集，请注意区分 Architecture（架构拓扑）和 Protocol（通信协议）——前者是逻辑上的连接方式，后者是技术上的通信标准。

### 原文标题：3 System-Level Communication

在进入具体的通信机制分析之前，作者首先定义了什么是"系统间通信"。系统间通信关注的是**整个多智能体系统作为一个整体的通信组织方式**。它回答的是以下高层问题：

- 系统中的 agent 是如何互相连接的？
- 通信的目的是什么？
- 系统使用什么技术标准来支持通信？

与系统内通信（Section 4）不同，系统间通信不关心单次消息的具体传递方式，而是关心通信的全局结构和组织逻辑。

#### 3.1 Architecture（架构）

作者将现有 LLM-MAS 的通信架构归纳为五种基础类型：

##### 3.1.1 Flat Architecture（扁平架构）

**定义**：所有 agent 处于同一层级，彼此之间可以直接通信。不存在中心协调者或层级结构。

**典型系统**：
- **ChatDev**：多个 agent（CEO、CTO、Programmer、Reviewer 等）在扁平结构中直接通信，通过自然语言对话协作开发软件；
- **Generative Agents**：25 个 agent 在一个模拟小镇中通过扁平的对等通信进行社交交互。

**优势**：
- 结构简单，易于实现和扩展；
- 每个 agent 可以直接获取所有其他 agent 的信息；
- 没有单点故障——任何一个 agent 失效，其他 agent 仍然可以直接通信。

**劣势**：
- 当 agent 数量增长时，通信复杂度呈 O(n^2) 增长；
- 缺乏全局协调，容易产生冲突和不一致；
- 每个 agent 需要处理来自所有其他 agent 的信息，信息过载风险高；
- 难以实施全局策略（如任务优先级、资源分配）。

> [!tip] 理解提示（非原文）
> 扁平架构适用于 agent 数量较少（通常 2-10 个）、任务相对简单、不需要强全局协调的场景。当 agent 增多到几十上百个时，纯粹扁平架构的信息过载和协调成本会变得不可接受。

##### 3.1.2 Hierarchical Architecture（层级架构）

**定义**：agent 按照层级组织，信息沿层级上下流动。通常有一个或多个上层 agent 负责决策和协调，下层 agent 负责执行。

**典型系统**：
- **AutoGen**：支持层级化的 agent 组织结构，manager agent 可以管理和调度多个 worker agent；
- **MetaGPT**：采用类公司的层级结构，从 CEO 到 Product Manager、Architect、Engineer，信息按层级流动；
- **AgentVerse**：支持多种组织结构，包括层级化的管理-执行模式。

**优势**：
- 分工明确，责任清晰；
- 上层 agent 可以进行全局决策和资源分配；
- 通信路径受控，降低了信息过载风险；
- 易于添加或移除下层 agent。

**劣势**：
- 上层 agent 构成单点故障和性能瓶颈；
- 层级延迟——信息需要经过多层传递；
- 下层 agent 的自主性受限；
- 层级结构的设计（几层、每层几个 agent）本身就是非平凡的设计决策。

##### 3.1.3 Team Architecture（团队架构）

**定义**：agent 被组织为多个相对独立的团队，每个团队内部有紧密的协作，团队之间通过指定的代表或协调者进行通信。

**典型系统**：
- **DyLAN**：动态组建 agent 团队，根据任务需求在运行时选择最优的 agent 组合；
- **AgentTeamwork**：将 agent 分组为功能团队（如前端团队、后端团队、测试团队），团队之间通过代表协调。

**优势**：
- 结合了扁平架构（团队内部）和层级架构（团队之间）的优点；
- 团队内部通信高效，成员之间紧密协作；
- 团队之间相对解耦，一个团队的变化不易影响其他团队；
- 支持专业化——每个团队可以专注于特定类型的任务。

**劣势**：
- 跨团队通信的协议和接口设计复杂；
- 需要决定 agent 到团队的分配策略（静态 vs. 动态）；
- 团队之间的协调可能成为新的瓶颈；
- 团队边界的定义本身可能限制信息的流动。

> [!tip] 理解提示（非原文）
> 团队架构特别适合大型复杂项目——比如软件开发中，前端团队、后端团队和测试团队各自独立运作，但需要定期同步。这种架构最接近真实人类组织的运作方式。

##### 3.1.4 Society Architecture（社会架构）

**定义**：大量 agent 在开放的社会结构中交互，没有预设的全局拓扑。agent 自主决定与谁通信、何时通信、通信什么内容。通信模式是涌现的而非预设的。

**典型系统**：
- **Generative Agents（大规模部署）**：在模拟小镇中，25 个 agent 自主形成社交关系，通信模式自然涌现；
- **Project Sid**：在 Minecraft 中部署上千个 agent，形成复杂的社会互动和涌现通信模式。

**优势**：
- 最接近真实人类社会，可以用于社会模拟和行为研究；
- 高度灵活，agent 自主适应环境和任务变化；
- 可能出现非预期的、有价值的涌现行为；
- 可扩展性理论上无上限。

**劣势**：
- 系统行为不可预测，难以调试和控制；
- 通信效率可能极低——大量 agent 在"说无关的话"；
- 难以引导系统向特定目标演进；
- 缺乏全局质量保证机制；
- 容易出现信息级联和群体幻觉。

##### 3.1.5 Hybrid Architecture（混合架构）

**定义**：根据任务需求或运行时上下文，动态切换或组合多种架构类型。

**典型特征**：
- **静态混合**：系统中不同部分采用不同的架构（如管理层采用层级架构，执行层采用团队架构）；
- **动态混合**：系统根据任务类型、agent 状态或上下文，在运行时切换架构模式。

**优势**：
- 灵活适配不同类型的任务和场景；
- 综合了各种架构的优点，规避单一架构的劣势；
- 理论上可以做到"针对每个任务选择最优架构"。

**劣势**：
- 设计复杂度最高——需要定义何时切换、如何切换、切换逻辑本身如何验证；
- 目前缺乏成熟的设计方法论和最佳实践；
- 架构切换可能引入通信中断和不一致。

##### 架构维度小结

作者通过一个对比表格总结了五种架构的核心特征：

| 架构类型 | 通信复杂度 | 全局协调能力 | 可扩展性 | 鲁棒性 | 适用场景 |
|----------|-----------|-------------|---------|--------|---------|
| Flat | 高（O(n^2)） | 弱 | 受限 | 高 | 小规模、简单任务 |
| Hierarchical | 中 | 强 | 中 | 中（上层脆弱） | 需要强协调的任务 |
| Team | 中 | 中 | 高 | 高 | 大型复杂项目 |
| Society | 不确定 | 极弱 | 理论上无限 | 高 | 社会模拟、涌现研究 |
| Hybrid | 取决于配置 | 取决于配置 | 取决于配置 | 取决于配置 | 需要灵活适配的场景 |

> [!note] 译者注（非原文）
> 这个表格是作者对五种架构的系统化对比。在实际选择架构时，通常需要在多个维度之间做权衡——没有一种架构在所有维度上都最优。

#### 3.2 Communication Goals（通信目标）

作者将 LLM-MAS 中的通信目标分为三种基本类型：

##### 3.2.1 Cooperation（合作）

**定义**：agent 为了实现共同目标而进行的信息共享、任务分配和互补协作。

**通信特征**：
- agent 之间共享信息和知识；
- agent 分工承担任务的不同部分；
- agent 之间互相补充各自的能力短板；
- 通信的目的是**最大化整体收益**。

**典型系统**：
- **ChatDev**：agent 合作完成软件开发；
- **MetaGPT**：agent 按照公司层级合作完成复杂项目；
- **AutoGen**：多个 agent 合作为用户完成任务。

**通信模式**：
- 信息广播：一个 agent 将重要发现告诉所有相关 agent；
- 请求-响应：一个 agent 向另一个 agent 请求信息或帮助；
- 任务委派：一个 agent 将子任务分配给最合适的 agent；
- 结果聚合：多个 agent 的结果被汇总成最终输出。

##### 3.2.2 Competition（竞争）

**定义**：agent 之间通过辩论、博弈或相互审查来实现更高质量的输出。

**通信特征**：
- agent 提出不同的观点或方案；
- agent 互相挑战和审查对方的输出；
- agent 在竞争中争取"胜出"；
- 通信的目的是**通过对抗来提升质量**。

**典型系统**：
- **Multi-Agent Debate**：多个 agent 就同一问题给出不同答案，通过辩论轮次达成共识或选出最优答案；
- **ChatEval**：多个 agent 竞争性地评估文本质量；
- **LLM-Debate**：使用辩论框架来改进推理质量。

**通信模式**：
- 辩论：agent 依次陈述观点、反驳、回应；
- 评审：一个 agent 评判另一个 agent 的输出；
- 投票：多个 agent 独立投票选出最优方案；
- 排名：agent 对多个候选方案进行排序。

> [!tip] 理解提示（非原文）
> 竞争型通信的一个关键价值在于：它利用 agent 之间的不一致性来发现和纠正错误。当多个 agent 独立审视同一问题并得出不同结论时，辩论过程能暴露每个观点中的薄弱环节。

##### 3.2.3 Mixed（混合）

**定义**：系统在合作和竞争之间动态切换或同时包含两种模式。

**通信特征**：
- 同一系统中的不同 agent 子群之间可能是竞争关系，子群内部是合作关系；
- 系统可能在不同阶段采用不同的通信目标（如先合作生成候选方案，再竞争筛选最优方案）；
- 通信目的的转换可能是预设的（规则驱动）或涌现的（agent 自主决定）。

**典型系统**：
- **DyLAN**：agent 在合作完成任务和竞争性评估之间动态切换；
- **MAD（Multi-Agent Debate）框架**：结合合作性信息共享和竞争性辩论。

**通信模式**：
- 阶段切换：先合作后竞争，或交替进行；
- 分组策略：组内合作、组间竞争；
- 角色互补：部分 agent 专职合作（如信息收集者），部分 agent 专职竞争（如批判者）。

##### 通信目标维度小结

作者指出，通信目标的选择深刻影响系统的行为特征：

| 通信目标 | 信息流动方向 | 典型失败模式 | 适用场景 |
|----------|-------------|-------------|---------|
| Cooperation | 多向共享 | 群体思维（groupthink）、信息级联 | 任务分解、互补协作 |
| Competition | 对立或并行 | 过度对抗、无法收敛 | 质量审查、推理验证 |
| Mixed | 取决于阶段 | 模式切换不当导致混乱 | 复杂多阶段任务 |

#### 3.3 Communication Protocols（通信协议）

作者指出，随着 LLM-MAS 从实验阶段走向工程落地，**通信协议的标准化**成为关键议题。本节回顾了三个新兴的 agent 通信协议：

##### 3.3.1 MCP（Model Context Protocol）

**提出者**：Anthropic（2024 年底）

**核心定位**：定义 LLM 与外部工具、数据源之间的标准化交互接口。

**关键特性**：
- **客户端-服务器架构**：MCP 服务器暴露工具和数据资源，MCP 客户端（LLM 应用）发现并调用这些资源；
- **标准化接口**：定义了统一的工具描述、调用、响应格式；
- **资源发现**：客户端可以自动发现服务器上可用的工具和数据；
- **安全模型**：支持权限控制和沙箱隔离。

**在 LLM-MAS 中的角色**：MCP 主要解决 agent 与外部世界的通信（工具、数据库、API），但它也为 agent 间通信提供了基础——如果一个 agent 通过 MCP 暴露自己的能力，其他 agent 就可以通过 MCP 来发现和调用这些能力。

##### 3.3.2 A2A（Agent-to-Agent Protocol）

**提出者**：Google（2025 年初）

**核心定位**：定义 agent 之间的标准化通信协议。

**关键特性**：
- **Agent Card**：每个 agent 通过一个"名片"声明自己的能力、接口和通信偏好；
- **任务导向通信**：通信围绕"任务"（task）组织，而非随意对话；
- **异步通信**：支持长运行任务的异步交互，agent 可以在等待结果时处理其他任务；
- **多模态支持**：支持文本、结构化数据、文件等多种内容格式；
- **安全与认证**：内置 agent 身份验证和授权机制。

**在 LLM-MAS 中的角色**：A2A 直接解决了 agent 之间"如何相互发现、如何发起通信、如何协调任务"的问题。它是目前最成熟的 agent 间通信协议规范。

##### 3.3.3 ANP（Agent Network Protocol）

**提出者**：开源社区（2025 年）

**核心定位**：去中心化的 agent 通信与协作协议。

**关键特性**：
- **去中心化**：不依赖中心化服务器或注册中心，agent 通过对等网络直接通信；
- **语义 Web 基础**：基于 W3C 标准（如 DID、Verifiable Credentials），利用语义 Web 技术实现 agent 的可发现性和互操作性；
- **自主身份**：每个 agent 拥有去中心化身份标识（DID）；
- **开放生态**：任何 agent 可以加入网络，无需中心化授权。

**在 LLM-MAS 中的角色**：ANP 适合需要开放、去中心化 agent 网络的场景（如大规模社会模拟、分布式问题求解），但其复杂性和缺乏成熟工具链也是现实的障碍。

##### 三种协议对比

作者通过一个对比表格帮助读者理解三者的定位差异：

| 维度 | MCP | A2A | ANP |
|------|-----|-----|-----|
| 核心目标 | LLM 与工具的交互 | Agent 间任务协作 | 去中心化 agent 网络 |
| 架构模式 | 客户端-服务器 | Agent-to-Agent（可中心化） | 完全去中心化 |
| 发现机制 | 客户端主动发现 | Agent Card 声明 | 去中心化 DID |
| 通信模式 | 同步请求-响应 | 同步 + 异步 | 异步消息 |
| 安全模型 | 权限控制 + 沙箱 | 身份认证 + 授权 | 自主身份 + VC |
| 成熟度 | 早期（快速增长） | 早期（Google 推动） | 极早期（社区驱动） |
| 互补关系 | 解决"agent 怎么用工具" | 解决"agent 之间怎么协作" | 解决"大规模 agent 怎么自组织" |

> [!note] 译者注（非原文）
> 这三个协议的出现标志着 LLM-MAS 领域的一个重要转折：从"每个系统自定义通信方式"走向"标准化通信基础设施"。作者在原文中并没有给出这么详细的对比表格，以上对比是译者基于原文描述和协议文档做的综合归纳。

## 本章小结

> [!note] 译者注（非原文）
> Section 3 建立了全文最重要的分析框架。三个要点值得记住：
>
> 1. **五种架构各有适用场景**：Flat 适合小规模、Hierarchical 适合强协调、Team 适合大项目、Society 适合社会模拟、Hybrid 最灵活但最难设计。
>
> 2. **通信目标不只是"合作"**：竞争和混合模式同样重要，尤其在质量审查和推理验证场景中，竞争性通信可以大幅提升输出质量。
>
> 3. **协议标准化正在进行中**：MCP、A2A、ANP 分别从不同角度解决 agent 通信的标准化问题，但它们目前都处于早期阶段，互操作性尚不成熟。

| ANP | Agent 网络协议 | 去中心化的 agent 通信协议 |
| Agent Card | Agent 名片 | A2A 中 agent 声明自身元信息的机制 |

## 思考题

### 题目 1：如果你要设计一个 100 个 agent 的系统来处理大规模代码审查，你会选择什么架构？为什么？

> [!tip] 理解提示（非原文）
> 考虑：100 个 agent 如果全部扁平连接，信息过载会有多严重？层级架构中，上层审查 agent 是不是新的瓶颈？

> [!note] 译者注（非原文）
> 推荐 Team 架构或 Hybrid 架构。将 100 个 agent 分组为多个专业团队（如语法审查团队、安全审查团队、性能审查团队、风格审查团队等），每个团队内部采用扁平或层级结构，团队之间通过代表协调。也可以考虑在审查的不同阶段切换架构——初步审查阶段用并行扁平结构（快速），深度分析阶段用层级结构（精确）。

### 题目 2：MCP、A2A 和 ANP 之间是什么关系？它们会互相竞争还是互补？

> [!warning] 易混点（非原文）
> 不要因为三个协议都叫"agent 通信协议"就认为它们解决同一个问题。

> [!note] 译者注（非原文）
> 三个协议解决的是不同层次的问题：MCP 解决的是 agent 与外部工具/数据的交互（纵向通信），A2A 解决的是 agent 之间的任务协作（横向通信），ANP 解决的是大规模去中心化 agent 网络中的发现和组织（网络层通信）。它们更可能是互补关系——一个完整的 LLM-MAS 可能同时使用 MCP（调用工具）、A2A（agent 间协作）和 ANP（大规模去中心化发现）。
