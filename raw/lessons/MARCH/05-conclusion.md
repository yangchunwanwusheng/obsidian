# MARCH — 5. 结论 (Conclusion) 与附录要点

> MARCH 建立了智能体共进化范式，为可扩展、高保真的 LLM 事实自我改进开辟了路径。

## 正文翻译

### 5 Conclusion

在本文中，我们引入了 MARCH，一个为缓解数据密集型 RAG 任务中的幻觉而设计的多智能体框架。通过将共享策略重构为包含 Solver、Proposer 和 Checker 的协调层级，MARCH 使 LLM 能够同时精炼其合成推理和独立锚定能力。

我们的实验证明，零容忍奖励机制与双轨迹 PPO 的结合有效地内化了一个自包含的验证循环，持续超越强基线。MARCH 在不同模型家族间的稳健泛化强调了其作为复杂知识领域中信息合成的可扩展、高保真解决方案的潜力。

我们相信这种智能体共进化范式为构建**内在可信和可验证**的大语言模型提供了一条有前景的路径。

> [!note] 译者注（非原文）
> "智能体共进化"（agentic co-evolution）是论文的核心叙事关键词。它强调的是：生成器和验证器在训练过程中**共同进化**，而非分别训练后简单组合。这种联合优化确保了生成能力与验证能力的协调发展。

## 附录要点

### A. 系统提示设计

论文附录 A 给出了三个智能体的完整系统提示：

**Solver（A.1）**：严格要求仅基于提供的文档回答，不引入外部知识或推断超出文档范围的信息。

**Proposer（A.2）**：以数据标注专家角色运行，对模型回答中的每个关键数字生成一个问答对。遵循两条规则：
1. **问题的清晰性和具体性**：每个问题必须包含足够上下文，确保数字是唯一正确答案
2. **答案格式**：必须是纯数字（整数或小数），不能包含百分比符号、"k"表示千、范围或文字

**Checker（A.3）**：以数据验证专家角色运行，基于参考材料回答问题。在回答前必须陈述证据来源。如果参考材料中找不到相关信息，必须声明"Cannot answer"。所有答案必须是纯数字。

> [!tip] 理解提示（非原文）
> 从系统提示可以看出，MARCH 当前实现主要聚焦于**数值型声明**的验证。Proposer 只提取数字相关的问答对，Checker 也只回答纯数字。这是论文的一个明确限制——对非数值型幻觉（如因果推理错误、定性描述偏差）的检测能力未覆盖。

### B.4 奖励作弊分析

MARCH 框架的一个潜在奖励作弊是：系统可能通过减少提出的问题数量来膨胀奖励，遵循"少说少漏"策略——较少的声明最小化了被审计为幻觉的概率。

实验确实观察到随着训练进行，所有配置中提出问题的平均数量逐渐下降。联合更新 Solver + Checker 的配置比仅 Solver 基线表现出更明显的下降趋势，表明 Checker 轨迹上的零容忍惩罚施加了更严格的一致性瓶颈，进一步激励 Proposer 修剪潜在风险声明。

然而，这种数量缩减可以通过简单的指令约束有效缓解。通过在提示中明确要求 Proposer 生成不少于 $k$ 个问题（如 $k=3$），问题密度的下降被显著遏制。带有约束的模型在整个自进化过程中保持更加稳定和更高的事实查询量。

### B.5 训练动态

分析显示：
- 准确率和奖励在 General 和 STEM 数据集上均呈现**稳定收敛**
- 联合 Solver + Checker 更新**始终优于**仅 Solver 配置
- 同时更新 Solver 和 Checker 两条轨迹在单步训练中**不会施加过多的计算负担**
- 累积训练时间在可接受范围内

### C. 案例研究

附录 C 提供了一个完整的训练步示例，展示了 Guadeloupe 帕金森综合征病例中各智能体的输入输出：

1. **Solver** 基于 8 篇文档生成回答，包含多个具体数值（52-59%、50%、89%、78% 等）
2. **Proposer** 提取 5 个数值型问答对
3. **Checker** 仅基于原始文档独立回答这 5 个问题，全部匹配 Solver 的声明
4. 因此 ZTR = 0（全部匹配），该轨迹获得正奖励

> [!note] 译者注（非原文）
> 这个案例清晰地展示了信息不对称的工作方式：Checker 从未看到 Solver 的回答内容，它只看到了 5 个具体问题和原始文档。Checker 必须独立从文档中找到证据并给出数字答案。如果 Solver 的某个数字是幻觉的，Checker 无法从文档中找到对应答案，两者不匹配，ZTR = -1。

## 整体评价

### 优势

1. **创新的框架设计**：信息不对称 + 三智能体分工 + 联合强化学习，从架构层面解决确认偏误
2. **自包含训练**：无需人工标注、无需外部工具，仅用查询和检索文档即可训练
3. **显著且一致的改进**：在多个基准、多个维度上大幅超越基座模型
4. **良好的正交互补性**：可与 RLHF、Few-Shot、CoT 等方法叠加
5. **模型无关**：在 Llama 和 Qwen 两个模型家族上都有效

### 局限

1. 仅覆盖数值型声明验证，定性/因果型幻觉未涉及
2. 主要在 8B 模型上验证，更大规模的扩展性未知
3. Checker 本身也是 8B 模型，审计能力有上限
4. 奖励作弊（减少问题数量）需要人工设计约束
5. 三次前向推理的计算开销在部署场景中的影响未讨论
