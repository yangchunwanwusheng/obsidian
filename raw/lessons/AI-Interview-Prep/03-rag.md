# RAG 检索增强生成——从原理到高级实践

> RAG 是 LLM 落地最实用的技术。面试核心：理解检索-生成的完整链路，能分析各种 RAG 变体的适用场景。

## 学习目标

- [ ] 理解 RAG 的核心架构与 Naive → Advanced → Modular 的演进
- [ ] 掌握 Embedding 模型选型与向量数据库对比
- [ ] 能设计 Chunk 策略和检索优化方案
- [ ] 理解 RAG vs Fine-tuning 的场景选择
- [ ] 能分析 RAG 系统的常见痛点并给出解决方案

## 一、为什么需要 RAG？

### 1.1 LLM 的三大痛点

| 痛点 | 表现 | RAG 如何解决 |
|------|------|-------------|
| **知识过时** | 训练数据截止后的事件一无所知 | 检索最新文档 |
| **幻觉** | 编造看似合理但不存在的事实 | 基于检索到的事实生成 |
| **领域知识不足** | 对公司内部文档/专业领域不了解 | 检索企业知识库 |

### 1.2 RAG 核心公式

```
RAG = 检索(Retrieval) + 增强(Augmented) + 生成(Generation)

用户提问
    │
    ▼
┌──────────┐     ┌──────────────┐
│ Query    │────▶│ Retriever    │
│ (用户问题)│     │ (检索器)     │
└──────────┘     └──────┬───────┘
                        │ Top-K 相关文档
                        ▼
                 ┌──────────────┐
                 │ Context      │
                 │ (检索结果)   │
                 └──────┬───────┘
                        │
                        ▼
                 ┌──────────────┐     ┌──────────┐
                 │ Generator    │────▶│ 最终回答  │
                 │ (LLM生成)    │     │          │
                 └──────────────┘     └──────────┘

Prompt = "基于以下参考资料回答问题：\n{context}\n\n问题：{query}"
```

## 二、RAG 架构演进

### 2.1 Naive RAG（基础版）

```
文档 → 分块(Chunk) → Embedding → 存入向量数据库
                                        │
用户问题 → Embedding → 向量相似度搜索 → 取Top-K → 拼接Prompt → LLM生成
```

**问题**：
- 检索质量不稳定（可能搜到不相关内容）
- Chunk 粒度不好把握
- 缺少对检索结果的评估

### 2.2 Advanced RAG（进阶版）

```
                 ┌─────────────────────────────────┐
                 │        Advanced RAG              │
                 │                                  │
索引阶段:        │  文档解析 → 语义分块 → 元数据标注  │
                 │       → 多级索引                  │
                 │                                  │
检索阶段:        │  Query 改写/扩展 → 混合检索       │
                 │       → 重排序(Rerank)           │
                 │       → 相关性过滤               │
                 │                                  │
生成阶段:        │  上下文压缩 → 答案生成            │
                 │       → 引用溯源                 │
                 └─────────────────────────────────┘
```

**Advanced RAG 的关键改进**：

| 阶段 | 改进点 | 效果 |
|------|--------|------|
| 索引 | 语义分块（按段落/标题切分） | Chunk 语义更完整 |
| 索引 | 元数据标注（来源/日期/权限） | 支持过滤检索 |
| 检索 | Query 改写（HyDE/多查询） | 提高召回率 |
| 检索 | 混合检索（向量+关键词） | 兼顾语义和精确匹配 |
| 检索 | Rerank 重排序 | 提高精度 |
| 生成 | 上下文压缩 | 减少 token 浪费 |
| 生成 | 引用溯源 | 可验证性 |

### 2.3 Modular RAG（模块化）

```
┌───────────────────────────────────────────────┐
│              Modular RAG（可插拔模块）          │
│                                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ 索引模块  │  │ 检索模块  │  │ 生成模块  │    │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │    │
│  │ │Parser│ │  │ │Search│ │  │ │LLM   │ │    │
│  │ │Chunker│ │  │ │Rerank│ │  │ │Compr.│ │    │
│  │ │Encoder│ │  │ │Filter│ │  │ │Verify│ │    │
│  │ └──────┘ │  │ └──────┘ │  │ └──────┘ │    │
│  └──────────┘  └──────────┘  └──────────┘    │
│                                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐    │

```

## 三、Embedding 模型与向量数据库

### 3.1 Embedding 模型选型

| 模型 | 维度 | 特点 | 适用场景 |
|------|------|------|---------|
| OpenAI text-embedding-3-large | 3072 | 效果好、贵 | 英文为主、预算充足 |
| BGE-large-zh-v1.5 | 1024 | 中文效果好、开源 | 中文场景首选 |
| M3E-large | 1024 | 中英双语、开源 | 中英混合 |
| GTE-Qwen2 | 任意(支持MRL) | 最新、多语言 | 追求最佳效果 |
| BGE-M3 | 1024 | 多语言多粒度 | 多语言企业场景 |

> [!tip] 面试场景：Embedding 模型怎么选？
> 1. **语言**：中文为主选 BGE/M3E，英文为主选 OpenAI/GTE
> 2. **成本**：预算有限选开源模型自己部署
> 3. **维度**：高维 = 表达力强但存储/计算贵，可用 MRL（Matryoshka）动态降维
> 4. **评估**：用 MTEB/C-MTEB 排行榜作参考

### 3.2 向量数据库对比

| 数据库 | 类型 | 特点 | 适用场景 |
|--------|------|------|---------|
| **Milvus** | 分布式 | 大规模、高性能、功能丰富 | 企业级生产环境 |
| **Chroma** | 嵌入式 | 轻量、Python 友好 | 原型开发、小规模 |
| **Pinecone** | 云服务 | 全托管、免运维 | 快速上线、不想运维 |
| **Weaviate** | 独立服务 | 支持混合搜索、GraphQL | 需要结构化+向量混合 |
| **Qdrant** | 独立服务 | Rust 写的，性能好 | 追求性能 |
| **pgvector** | PG 扩展 | 复用 PostgreSQL | 已有 PG 基础设施 |
| **FAISS** | 库 | Meta 开源、纯库 | 不需要服务化 |

> [!hint]- 面试场景：生产环境选哪个向量数据库？
> **小规模（<100万向量）**：Chroma/Qdrant（简单好用）
> **中大规模（100万-10亿）**：Milvus（分布式、成熟）
> **已有 PG 基础设施**：pgvector（最小运维成本）
> **不想运维**：Pinecone/Zilliz Cloud（全托管）
> **关键考量**：是否需要过滤检索（元数据+向量混合）、是否需要分布式、QPS 要求

## 四、Chunk 策略——面试高频

### 4.1 常见分块策略

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定大小 | 每 512 token 切一块 | 简单 | 可能切断语义 |
| 递归字符 | 按 \n\n → \n → 句号 递归切 | 保留段落结构 | 粒度不均匀 |
| 语义分块 | Embedding 相似度变化点切 | 语义最完整 | 计算量大 |
| 文档结构 | 按标题/章节/页切 | 结构清晰 | 依赖文档格式 |
| 句子窗口 | 检索时返回前后文 | 上下文完整 | 存储冗余 |

### 4.2 Chunk 大小怎么选？

```
Chunk 太小（<100 token）:
  ✗ 语义不完整："这个方法的核心优势在于" → 缺少下文
  ✗ 检索噪声高

Chunk 太大（>1000 token）:
  ✗ 检索精度下降（大块中只有小部分相关）
  ✗ 浪费 LLM 上下文窗口

推荐范围：
  - QA 场景：256-512 token
  - 长文档摘要：512-1024 token
  - 代码检索：按函数/类切分
  - 最佳实践：chunk_size=512, overlap=50-100
```

### 4.3 元数据增强

```python
# 为每个 chunk 附加元数据
chunk_metadata = {
    "source": "company_handbook_2024.pdf",
    "page": 42,
    "section": "第三章-薪资福利",
    "author": "HR部门",
    "date": "2024-01-15",
    "access_level": "internal"
}

# 检索时可过滤
results = vector_db.search(
    query="年假有几天",
    filter={"section": "第三章-薪资福利", "access_level": "internal"}
)
```

## 五、检索优化——面试加分项

### 5.1 Query 改写

```
原始Query: "公司报销流程"
         │
         ▼
┌─────────────────────────────────────────┐
│  HyDE (Hypothetical Document Embedding) │
│  让 LLM 先生成假设性回答，用回答做检索   │
│                                         │
│  LLM 生成: "公司报销流程如下：           │
│  1. 登录OA系统... 2. 填写报销单..."     │
│         │                               │
│         ▼                               │
│  用这个假设回答的 Embedding 去检索       │
│  → 比原始 query 更接近真实文档          │
└─────────────────────────────────────────┘

Multi-Query（多查询扩展）:
  原始: "公司报销流程"
  扩展: ["如何提交报销申请", "报销审批流程", "差旅报销标准"]
  → 分别检索，合并结果
```

### 5.2 混合检索

```
┌──────────────┐     ┌──────────────┐
│  向量检索     │     │  关键词检索   │
│ (语义相似度)  │     │  (BM25/TF-IDF)│
│  Top-K₁      │     │  Top-K₂      │
└──────┬───────┘     └──────┬───────┘
       │                    │
       ▼                    ▼
┌──────────────────────────────────┐
│         RRF (Reciprocal Rank     │
│         Fusion) 融合排序         │
└──────────────┬───────────────────┘
               │
               ▼
         合并去重后的 Top-K
```

**为什么需要混合检索？**
- 向量检索擅长语义理解，但可能漏掉精确匹配（如产品编号、人名）
- 关键词检索擅长精确匹配，但理解不了语义
- 混合检索兼顾两者

### 5.3 Rerank（重排序）

```
初始检索 Top-20
       │
       ▼
┌──────────────────┐
│ Cross-Encoder     │  ← 比 Bi-Encoder 更精确但更慢
│ Rerank 模型       │  ← 对每个 (query, doc) 对打分
│ (如 bge-reranker) │
└────────┬─────────┘
         │
         ▼
重排序后 Top-5 → 送入 LLM
```

> [!success]- 面试场景：Bi-Encoder vs Cross-Encoder？
> **Bi-Encoder**（Embedding 检索）：
> - Query 和 Document 分别编码成向量，计算余弦相似度
> - 速度快（可预计算文档向量）O(1) 检索
> - 但精度有限（编码时不知道 query 内容）
>
> **Cross-Encoder**（Rerank）：
> - 把 Query 和 Document 拼接后一起编码
> - 精度高（能交叉注意力）
> - 但速度慢（每个文档都要过一遍模型）
>
> **最佳实践**：先用 Bi-Encoder 粗筛 Top-100，再用 Cross-Encoder 精排 Top-5

## 六、RAG vs Fine-tuning

| 维度 | RAG | Fine-tuning |
|------|-----|------------|
| 知识更新 | ✅ 实时更新文档即可 | ❌ 需要重新训练 |
| 成本 | 低（检索+推理） | 高（训练算力） |
| 可解释性 | ✅ 可追溯来源 | ❌ 黑盒 |
| 适用数据 | 外部知识/文档 | 行为模式/风格 |
| 幻觉控制 | ✅ 基于事实生成 | ⚠️ 仍可能幻觉 |
| 实时性 | ✅ 最新信息 | ❌ 截止训练数据 |

> [!hint]- 面试场景：什么时候用 RAG，什么时候用 Fine-tuning？
> **用 RAG**：知识密集型问答、需要最新信息、需要引用来源
> **用 Fine-tuning**：改变模型输出风格/格式、特定领域术语、简化 prompt
> **两者结合**：先 Fine-tune 模型理解领域，再用 RAG 补充具体知识

## 七、RAG 评估

### 7.1 RAGAS 评估框架

| 指标 | 评估什么 | 计算方式 |
|------|---------|---------|
| **Faithfulness** | 生成答案是否忠于检索内容 | 答案中的每个声明是否可溯源 |
| **Answer Relevancy** | 答案是否切题 | 答案与问题的相关性 |
| **Context Precision** | 检索结果中有用内容的比例 | 检索文档中相关文档的排名 |
| **Context Recall** | 是否检索到了所有相关信息 | Ground truth 是否被检索覆盖 |

## 八、面试高频问题

> [!success]- Q1：RAG 系统的检索结果不相关怎么办？
> 1. **检查 Embedding 模型**：模型是否适合你的语言/领域
> 2. **优化 Chunk 策略**：调整大小和 overlap
> 3. **加 Query 改写**：HyDE/Multi-Query 扩展
> 4. **混合检索**：向量 + BM25
> 5. **加 Rerank**：Cross-Encoder 精排
> 6. **加元数据过滤**：缩小检索范围

> [!success]- Q2：如何处理 RAG 中的"中间信息丢失"问题（Lost in the Middle）？
> LLM 倾向于关注 context 开头和结尾的内容，中间容易被忽略。
> 1. **减少无关文档**：只传真正相关的 Top-3，而非 Top-10
> 2. **重排序**：最相关的放开头和结尾
> 3. **文档压缩**：只保留与问题相关的部分
> 4. **分批处理**：多个文档分别生成答案，再汇总

> [!success]- Q3：设计一个企业级 RAG 系统，需要考虑什么？
> 1. **数据安全**：权限控制（不同用户看到不同文档）
> 2. **多模态**：支持 PDF/Word/表格/图片
> 3. **实时更新**：文档变更后自动重新索引
> 4. **多语言**：中英混合检索
> 5. **评估体系**：持续监控检索和生成质量
> 6. **缓存策略**：高频问题缓存答案
> 7. **可追溯性**：每个回答标注引用来源

## 关键要点回顾

1. **RAG = 检索 + 增强 + 生成**，解决 LLM 的知识过时、幻觉、领域知识不足
2. **Chunk 策略**是 RAG 的基础：256-512 token + 合理 overlap + 元数据
3. **混合检索 + Rerank** 是当前最佳实践
4. **向量数据库选型**要看规模、是否需要分布式、是否需要混合查询
5. **RAG 评估**要用 RAGAS 框架持续监控质量
