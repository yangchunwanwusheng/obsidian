# 业界热点 Agent 系统深度分析——从 Claude Code 到 Manus

> 掌握业界最新 Agent 系统的架构设计，是面试中证明技术视野的最佳方式。本篇深度拆解 Claude Code、Cursor、Manus、OpenHands 的核心设计。

## 学习目标

- [ ] 能深入分析 Claude Code 的 Agent 循环、工具系统、多智能体架构
- [ ] 能对比 Cursor / Windsurf / Claude Code 的设计差异
- [ ] 理解 Manus 的多 Agent 并行架构
- [ ] 掌握 MCP 协议的设计动机和架构
- [ ] 能在面试中自信回答"设计一个 Agent 系统"的开放题

## 一、为什么面试要考察 Agent 热点系统？

```
面试官的思维链：

"你了解 Claude Code 吗？"
  → 了解 → "说说它的 Agent 循环是怎么工作的" → 考察深度
  → 了解 → "它和 Cursor 有什么区别？" → 考察对比分析能力
  → 不了解 → "那你了解什么 Agent 系统？" → 考察视野
  → 都不了解 → 只能考 ReAct 理论（不够加分）

信号：能说清楚"为什么 Claude Code 这样设计"的候选人，技术视野 > 80% 竞争者
```

### 2.2 工具系统设计

Claude Code 的工具系统是其核心差异化能力：

```python
# 工具定义格式（简化版）
tools = {
    # ── 文件操作 ──
    "Read": {
        "description": "读取文件内容，支持行号范围",
        "params": {"file_path": "str", "offset": "int?", "limit": "int?"},
        "permission": "auto"  # 自动允许
    },
    "Write": {
        "description": "创建新文件或完整覆盖",
        "params": {"file_path": "str", "content": "str"},
        "permission": "confirm"  # 需要用户确认
    },
    "Edit": {
        "description": "基于字符串匹配的精确编辑（非整文件覆盖）",
        "params": {
            "file_path": "str",
            "old_string": "str",  # 精确匹配要替换的内容
            "new_string": "str"   # 替换后的内容
        },
        "permission": "confirm"
    },

    # ── 搜索工具 ──
    "Grep": {
        "description": "正则搜索文件内容（基于 ripgrep）",
        "params": {"pattern": "str", "path": "str?", "glob": "str?"}
    },
    "Glob": {
        "description": "文件名模式匹配搜索",
        "params": {"pattern": "str", "path": "str?"}
    },

    # ── 执行工具 ──
    "Bash": {
        "description": "在沙箱终端执行命令",
        "params": {
            "command": "str",
            "timeout": "int?",      # 最大 600s
            "run_in_background": "bool?"  # 异步执行
        },
        "permission": "confirm"  # 危险命令需确认
    },

    # ── 子智能体 ──
    "Agent": {
        "description": "启动子智能体执行独立任务",
        "params": {
            "prompt": "str",           # 子智能体的任务描述
            "subagent_type": "str?",   # 特化类型
            "run_in_background": "bool?"
        }
    },

    # ── 浏览器 ──
    "browser_navigate": { "..." : "Playwright 浏览器操作" },
    "browser_click": { "..." : "点击元素" },
    "browser_snapshot": { "..." : "获取页面快照" },
}
```

**关键设计决策**：

| 设计 | 原因 | 替代方案 |
|------|------|---------|
| Edit 用字符串匹配而非行号 | Git diff 后行号可能变化 | 行号编辑（不稳定） |
| Bash 需要 confirm | 防止误删文件 | 全自动（危险） |
| Grep/Glob 内置而非 Bash | 更精确的权限控制 | `!grep xxx`（绕过权限） |
| 子智能体有上下文隔离 | 防止子任务污染主上下文 | 共享完整上下文（浪费 token） |

### 2.3 Agent 循环详解

```
Claude Code 的单次 Agent 循环：

┌──────────┐
│ 用户输入  │ "帮我重构这个函数，增加错误处理"
└────┬─────┘
     ▼
┌──────────────────────────────────────┐
│ 1. 上下文构建                         │
│    [CLAUDE.md全局] + [CLAUDE.md项目]  │
│    + [对话历史] + [用户输入]           │
│    → 总 Token 预算分配               │
└──────────────────┬───────────────────┘
                   ▼
┌──────────────────────────────────────┐
│ 2. LLM 推理（Think）                  │
│    模型生成：                          │
│    Thought: 我需要先读取文件内容...    │
│    Action: Read(file_path="xxx")      │
└──────────────────┬───────────────────┘
                   ▼
┌──────────────────────────────────────┐
│ 3. 工具执行（Act）                     │
│    系统执行 Read，返回文件内容          │
│    Observation: "def foo(): ..."      │
└──────────────────┬───────────────────┘
                   ▼
┌──────────────────────────────────────┐
│ 4. 继续推理（Observe → Think）        │
│    LLM 看到文件内容，决定下一步        │
│    Thought: 好的，我需要用 Edit ...    │
│    Action: Edit(file_path, old, new)  │
└──────────────────┬───────────────────┘
                   ▼
         ... 重复 Think/Act/Observe ...
                   │
                   ▼
┌──────────────────────────────────────┐
│ 5. 终止条件                           │
│    - 任务完成 → 输出最终回答           │
│    - 达到最大步数 → 强制停止           │
│    - Token 预算耗尽 → 压缩上下文      │
│    - 用户中断 → Ctrl+C               │
└──────────────────────────────────────┘
```

### 2.4 多智能体架构

```
主 Agent
  │
  ├── 启动子 Agent A（前景模式）
  │     └── 独立上下文，完成后结果回传主 Agent
  │
  ├── 启动子 Agent B（后台模式 run_in_background）
  │     └── 异步执行，完成后通知主 Agent
  │
  └── 启动子 Agent C（工作树隔离 worktree）
        └── 在独立 Git worktree 中工作，互不干扰

关键设计：
1. 子 Agent 的上下文是隔离的（不会撑爆主上下文）
2. 结果通过 tool_result 回传（结构化）
3. 后台任务不阻塞主 Agent
4. 并发控制：防止过多子 Agent 同时运行
```

### 2.5 权限与安全系统

```
权限三级模型：

Level 1: 自动允许（auto）
  - Read 文件
  - Grep/Glob 搜索
  - 子智能体启动

Level 2: 需要确认（confirm）
  - Write/Eidt 文件修改
  - Bash 命令执行
  - MCP 工具调用

Level 3: 禁止（deny）
  - 删除 .git 目录
  - 修改 CLAUDE.md 中的安全规则
  - 访问敏感文件（.env, credentials）

安全机制：
- Edit 工具的 old_string 必须唯一匹配（防止误改）
- Bash 命令有超时限制（防止无限运行）
- 子智能体有独立权限空间
- 用户可随时 Ctrl+C 中断
```

### 2.6 MCP（Model Context Protocol）

```
MCP 架构：

┌──────────────┐         ┌──────────────┐
│   MCP Host   │         │  MCP Server  │
│  (Claude Code│◄───────►│  (第三方工具) │
│   等 AI 应用) │  STDIO/ │              │
│              │  HTTP    │  Tools       │
│              │  协议    │  Resources   │
│              │         │  Prompts     │
└──────────────┘         └──────────────┘

MCP 解决的问题：
1. 标准化工具接口 → 不同 AI 应用可以复用同一工具
2. 安全隔离 → 工具在独立进程中运行
3. 动态发现 → 运行时发现可用工具，不需要硬编码

面试关键点：
- MCP ≠ Function Calling（MCP 是协议，FC 是 LLM 能力）
- MCP Server 是独立的进程（隔离性）
- Claude Code 通过 MCP Client 连接 Server
```

> [!hint]- 面试场景：Claude Code 的 Agent 循环和标准 ReAct 有什么区别？
> 1. **工具更丰富**：不只是简单 API 调用，还有文件编辑、代码搜索、子智能体调度
> 2. **权限系统**：三级权限模型，不是所有操作都能自动执行
> 3. **多智能体**：主 Agent 可以启动子 Agent 并行处理
> 4. **持久记忆**：通过 CLAUDE.md + MEMORY.md 跨会话保持状态
> 5. **上下文压缩**：对话过长时自动 compaction，而非简单截断
> 6. **安全边界**：Edit 的 old_string 唯一匹配、Bash 超时、文件保护

> [!hint]- 面试场景：Claude Code 如何保证代码修改的安全性？
> 1. **精确编辑**：Edit 用 old_string 精确匹配，只改该改的地方
> 2. **权限分级**：文件修改需要 confirm，不会静默改文件
> 3. **Git 保护**：每次修改在 Git 跟踪下，可随时回滚
> 4. **沙箱执行**：Bash 命令在受限环境中运行
> 5. **用户中断**：任何时候 Ctrl+C 可停止
> 6. **敏感文件保护**：`.env`、credentials 等自动拦截

### 3.3 三方对比：Cursor vs Windsurf vs Claude Code

| 维度 | Claude Code | Cursor | Windsurf |
|------|------------|--------|----------|
| **形态** | CLI 终端 Agent | IDE 内嵌（VS Code Fork） | IDE 内嵌（VS Code Fork） |
| **上下文管理** | 自动 + CLAUDE.md | 手动 @file/@codebase | 自动全局感知 |
| **编辑方式** | Edit (字符串匹配) | Diff-based 补丁 | Diff-based 补丁 |
| **多文件** | 主 Agent 协调 | Composer Agent | Cascade Agent |
| **记忆系统** | CLAUDE.md + MEMORY.md | .cursorrules | 规则 + 持久记忆 |
| **工具生态** | MCP 协议扩展 | 内置 + 插件 | 内置 |
| **模型支持** | Claude 系列 | Claude/GPT/Gemini | Claude/GPT/Gemini |
| **安全机制** | 三级权限 + Git 保护 | IDE 级别确认 | IDE 级别确认 |
| **最擅长** | 自动化脚本/DevOps | 全栈大型项目 | 多文件重构 |
| **学习曲线** | 中（需要了解 CLI） | 低（IDE 原生体验） | 低（IDE 原生体验） |
| **适用团队** | 个人开发者/DevOps | 全栈团队 | 全栈团队 |

### 3.4 共同的架构模式

```
三者共享的核心模式：

1. Agent 循环：Think → Act → Observe（ReAct 变体）
2. 上下文窗口管理：Token 预算 + 压缩
3. 代码索引：向量数据库 + 语义搜索
4. Diff 编辑：精确修改而非整文件替换
5. 权限控制：用户确认敏感操作

三者的核心差异在于"形态"和"上下文获取方式"：
- Claude Code：CLI 形态，依赖用户输入和文件搜索
- Cursor：IDE 形态，依赖手动 @ 引用
- Windsurf：IDE 形态，自动感知开发者操作
```

> [!hint]- 面试场景：Cursor 和 Claude Code 的核心架构差异是什么？
> 1. **形态差异**：Cursor 是 IDE 内嵌（有 GUI），Claude Code 是 CLI 终端（纯文本交互）
> 2. **上下文获取**：Cursor 靠 @file 手动引用，Claude Code 靠 Grep/Glob 自动搜索
> 3. **编辑策略**：Cursor 用 Diff 补丁（语义级），Claude Code 用字符串匹配（文本级）
> 4. **扩展性**：Claude Code 通过 MCP 协议扩展，Cursor 通过内置工具+插件
> 5. **适用场景**：Cursor 适合大型 IDE 项目，Claude Code 适合自动化/DevOps/脚本
> **共同点**：都基于 ReAct 循环 + LLM 推理 + 工具调用 + Token 管理

```
          │   任务分解引擎    │
          │   生成 100 个     │
          │   子任务          │
          └────────┬─────────┘
                   │
     ┌──────┬──────┼──────┬──────┐
     ▼      ▼      ▼      ▼      ▼
  Agent1  Agent2  Agent3  ...  Agent100
  (美国)  (欧洲)  (中国)       (日本)
     │      │      │            │
     ▼      ▼      ▼            ▼
  搜索+   搜索+   搜索+       搜索+
  分析    分析    分析        分析
     │      │      │            │
     └──────┴──────┼──────┴──────┘
                   ▼
          ┌──────────────────┐
          │  汇总 Agent       │
          │  整合所有结果      │
          │  生成最终报告      │
          └──────────────────┘

关键设计：
1. 每个 Agent 在独立沙箱中运行
2. 中间结果以文件形式共享
3. 用 Markdown 管理任务规划（结构化）
4. 多模型混合调用（不同任务选不同模型）
```

### 4.3 Manus 的启示与争议

```
启示：
1. "交付型 Agent"是正确方向——用户要的是结果，不是建议
2. 虚拟机沙箱是通用 Agent 的必要条件
3. 多 Agent 并行是处理复杂任务的有效策略

争议与教训：
1. Demo 效果 ≠ 真实体验（130天后的用户落差）
2. 成本问题：100 个 Agent 并行 = 巨额 Token 消耗
3. "Less Structure" 的边界在哪？太自动化用户会失控
4. 通用 Agent 的可靠性仍是核心挑战

面试加分点：
"我认为 Manus 的核心贡献是证明了'交付型 Agent'的可行性，
但它的商业化困境也说明，Agent 的可靠性比功能丰富更重要。"
```

> [!hint]- 面试场景：Manus 的多 Agent 并行和传统主从模式有什么区别？
> **传统主从模式**：主 Agent 串行分配任务给子 Agent，子 Agent 结果回传后主 Agent 再分配下一个
> **Manus 并行模式**：
> 1. 一次性分解为所有子任务（DAG 结构）
> 2. 无依赖的子任务并行启动（100 个同时）
> 3. 有依赖的子任务等待前置完成
> 4. 最后汇总 Agent 整合所有结果
> **优势**：大幅减少总执行时间（并行度更高）
> **劣势**：成本高、协调复杂、中间结果管理困难

### 5.2 关键设计决策

| 设计决策 | 原因 | 替代方案 |
|---------|------|---------|
| Docker 沙箱 | 完全隔离，安全 | 直接执行（危险） |
| 事件驱动 | 解耦，可扩展 | 轮询（低效） |
| Agent SDK | 可插拔不同 Agent | 硬编码（不灵活） |
| 多模型支持 | 用户自由选择 | 绑定单一模型 |

> [!hint]- 面试场景：OpenHands 为什么选择 Docker 沙箱？有什么替代方案？
> **选择 Docker 的原因**：
> 1. **完全隔离**：文件系统、网络、进程全部隔离
> 2. **可重现**：每次任务从干净环境开始
> 3. **资源限制**：可限制 CPU/内存/磁盘
> 4. **快照回滚**：可保存和恢复容器状态
>
> **替代方案**：
> 1. **gVisor**：更轻量的沙箱，但兼容性差
> 2. **Firecracker microVM**：AWS 的轻量虚拟机，启动更快
> 3. **WebAssembly (WASM)**：最轻量但功能受限
> 4. **进程级沙箱 (seccomp/bpf)**：最轻但隔离最弱

## 六、面试高频问题汇总

> [!success]- Q1：Claude Code 的 Agent 循环具体是怎么工作的？
> 1. 接收用户输入，构建上下文（CLAUDE.md + 对话历史 + 系统提示）
> 2. LLM 推理决定下一步行动（Think → 选择工具 → 调用）
> 3. 执行工具，获取观察结果
> 4. 判断是否需要继续（未完成 → 回到第2步）
> 5. 任务完成 → 生成最终回答
> **关键差异**：不是简单的 ReAct，还有权限管理、上下文压缩、子智能体调度

> [!success]- Q2：Cursor 和 Claude Code 的架构差异是什么？
> 见上方 3.3 节的详细对比表格。核心差异在形态（IDE vs CLI）、上下文获取方式（手动@ vs 自动搜索）、编辑策略（Diff补丁 vs 字符串匹配）

> [!success]- Q3：如何设计一个安全的代码执行 Agent？
> 1. **沙箱隔离**：所有代码在 Docker/VM 中执行
> 2. **权限分级**：读取自动、修改确认、删除禁止
> 3. **超时保护**：每个命令有执行时间上限
> 4. **审计日志**：所有操作有据可查
> 5. **Git 保护**：每次修改可回滚
> 6. **敏感文件过滤**：`.env`、密钥文件自动拦截

> [!success]- Q4：MCP 协议解决了什么问题？
> 1. **标准化**：不同 AI 应用（Claude Code / IDE / Web）可以复用同一工具
> 2. **隔离性**：工具在独立进程中运行，不影响主应用
> 3. **动态发现**：运行时发现可用工具，不需要硬编码
> 4. **生态**：第三方开发者可以开发 MCP Server，扩展 AI 能力

> [!success]- Q5：如果让你从零设计一个编程 Agent，你会怎么做？
> **架构**：
> 1. **Agent 循环**：ReAct 模式 + 最大步数限制
> 2. **工具系统**：文件读写 + 代码搜索 + 终端执行 + 子 Agent
> 3. **记忆系统**：CLAUDE.md 项目规则 + 自动记忆笔记
> 4. **安全**：Docker 沙箱 + 权限分级 + Git 保护
> 5. **评估**：任务完成率 + 代码质量 + Token 效率

> [!success]- Q6：Manus 的"交付型 Agent"和"建议型 Agent"有什么区别？
> **建议型**（如 ChatGPT）：只给文字建议，用户自己去执行
> **交付型**（如 Manus）：直接执行任务并交付最终结果
> **核心差异**：交付型需要工具执行能力 + 沙箱环境 + 任务验证
> **权衡**：交付型更高效但更不可控，建议型更安全但效率低

> [!success]- Q7：Agent 系统的 Token 成本如何控制？
> 1. **上下文压缩**：长对话自动 compaction
> 2. **子智能体隔离**：子任务的上下文不污染主对话
> 3. **工具优化**：Grep 代替读取整个文件
> 4. **模型分层**：简单任务用小模型，复杂任务用大模型
> 5. **缓存**：相同 prompt + context 的结果缓存
> 6. **Token 预算**：硬性限制每轮的最大 token 消耗

> [!success]- Q8：OpenHands 的 Agent SDK 设计有什么好处？
> 1. **可插拔**：轻松替换不同 Agent 策略（CodeAct/Planner/Critic）
> 2. **可扩展**：开发者可以自定义 Agent 模板
> 3. **可组合**：多个 Agent 可以串联或并联
> 4. **可测试**：每个 Agent 独立测试，互不干扰
> 5. **模型无关**：Agent 逻辑和 LLM 选择解耦

## 关键要点回顾

1. **Claude Code** = CLI Agent + 三层记忆（CLAUDE.md）+ MCP 扩展 + 多智能体调度
2. **Cursor** = IDE Agent + 向量索引 + @引用 + Composer 多文件编辑
3. **Manus** = 通用 Agent + 虚拟机沙箱 + 多 Agent 并行 + "交付型"理念
4. **OpenHands** = 开源 Agent SDK + Docker 沙箱 + 事件驱动架构
5. **共同模式**：ReAct 循环 + 工具系统 + 记忆 + 安全机制 + Token 管理
6. **面试策略**：能说清楚"为什么这样设计"比记住架构更重要
