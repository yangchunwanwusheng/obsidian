# AI Memory 记忆系统——让 LLM "记住"一切

> Memory 是 Agent 的关键组件。面试重点：理解短期记忆、长期记忆的设计模式，以及上下文窗口管理策略。

## 学习目标

- [ ] 理解 AI 记忆的分类体系（短期/长期/工作记忆）
- [ ] 掌握对话历史管理的关键技术
- [ ] 能设计一个实用的记忆系统架构
- [ ] 理解记忆压缩、遗忘、检索策略

## 一、为什么 AI 需要记忆？

### 1.1 问题场景

```
用户（第1轮）: "我叫张三，我在做推荐系统项目"
AI: "你好张三！推荐系统很有趣..."

用户（第10轮）: "根据我之前说的项目，帮我设计一个方案"
AI（无记忆）: "请问你之前提到了什么项目？"  ← 糟糕！
AI（有记忆）: "根据你的推荐系统项目，我建议..." ← 好！
```

### 1.2 记忆分类体系

```
                ┌────────────────────────┐
                │    AI Memory System    │
                └───────────┬────────────┘
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                  ▼
   ┌──────────────┐ ┌──────────────┐  ┌──────────────┐
   │  短期记忆     │ │  工作记忆     │  │  长期记忆     │
   │ Short-Term   │ │ Working      │  │ Long-Term    │
   ├──────────────┤ ├──────────────┤  ├──────────────┤
   │ 当前对话历史  │ │ 当前任务状态  │  │ 用户画像     │
   │ 上下文窗口内  │ │ 执行计划     │  │ 历史对话摘要  │
   │ 最近的交互   │ │ 中间结果     │  │ 知识库       │
   └──────────────┘ └──────────────┘  └──────────────┘
          │                 │                  │
          ▼                 ▼                  ▼
    对话上下文窗口    Agent State       向量数据库/摘要
    (8K-128K tokens)  (structured)      (persistent)
```

## 二、短期记忆——对话历史管理

### 2.1 核心挑战：上下文窗口有限

```
LLM 上下文窗口（Context Window）:
├── GPT-4: 128K tokens
├── Claude-3: 200K tokens
├── Llama-3: 128K tokens
└── 实际可用: 减去 prompt + 检索内容 + 输出空间

问题: 对话 50+ 轮后，历史消息可能超出窗口！
```

### 2.2 对话历史管理策略

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **截断** | 只保留最近 N 轮 | 简单 | 丢失早期上下文 |
| **摘要** | 用 LLM 压缩早期对话 | 保留关键信息 | 摘要可能丢失细节 |
| **滑动窗口** | 保留最近 K 轮 + 对早期摘要 | 平衡 | 实现稍复杂 |
| **Token 预算** | 按重要性分配 token 预算 | 灵活 | 需要重要性评估 |
| **混合** | 最近几轮完整 + 早期摘要 + 长期记忆检索 | 最优 | 最复杂 |

### 2.3 对话摘要实现

```python
class ConversationMemory:
    def __init__(self, max_recent_turns=5, max_summary_length=500):
        self.recent_messages = []  # 最近的完整消息
        self.summary = ""          # 早期对话的摘要
        self.max_recent_turns = max_recent_turns
        self.max_summary_length = max_summary_length

    def add_message(self, role, content):
        self.recent_messages.append({"role": role, "content": content})

        # 当消息数超过阈值，触发摘要
        if len(self.recent_messages) > self.max_recent_turns * 2:
            self._compress_history()

    def _compress_history(self):
        """把较早的消息压缩成摘要"""
        old_messages = self.recent_messages[:-self.max_recent_turns * 2]
        new_summary = llm_summarize(old_messages, self.summary)
        self.summary = new_summary
        self.recent_messages = self.recent_messages[-self.max_recent_turns * 2:]

    def get_context(self):
        """返回当前完整上下文"""
        context = []
        if self.summary:
            context.append({
                "role": "system",
                "content": f"[对话历史摘要]\n{self.summary}"
            })
        context.extend(self.recent_messages)
        return context
```

> [!hint]- 面试场景：对话 100 轮后，用户引用了第 5 轮的细节，怎么办？
> 1. **摘要中保留关键实体**（人名、项目名、约定等）
> 2. **长期记忆检索**：将历史对话存入向量数据库，按相关性检索
> 3. **实体追踪**：维护一个实体表，记录用户提到的关键信息
> 4. **主动提醒**：当摘要丢失细节时，主动告诉用户"我记得你之前提到过..."

### 3.2 MemGPT / Letta 架构

MemGPT（Memory-GPT）是当前最有影响力的记忆架构：

```
核心思想：借鉴操作系统的虚拟内存管理

┌────────────────────────────────────┐
│          Main Context              │  ← 相当于 RAM
│   (LLM 上下文窗口内的信息)         │     容量有限
│   ┌─────┐┌─────┐┌─────┐           │
│   │系统  ││对话  ││工作  │          │
│   │指令  ││历史  ││记忆  │          │
│   └─────┘└─────┘└─────┘           │
└───────────────┬────────────────────┘
                │ Page In/Out
┌───────────────▼────────────────────┐
│          External Memory           │  ← 相当于 硬盘
│   (持久化存储，上下文窗口外)        │     容量大
│   ┌─────────────────────────┐     │
│   │ 对话历史 (全量)          │     │
│   │ 用户画像 (结构化)        │     │
│   │ 知识片段 (向量索引)      │     │
│   │ 任务状态 (JSON)          │     │
│   └─────────────────────────┘     │
└────────────────────────────────────┘
```

**关键操作**：
- **Page In**：从外部记忆检索相关信息，塞入上下文窗口
- **Page Out**：把上下文窗口中的信息摘要/存储到外部记忆
- **自主管理**：LLM 自己决定什么时候读/写记忆（通过 function calling）

### 3.3 记忆写入策略

```python
# 哪些信息值得存入长期记忆？
def should_memorize(message: str, context: dict) -> bool:
    """
    判断一条消息是否值得写入长期记忆
    """
    # 1. 包含个人信息的（姓名、偏好、背景）
    if contains_personal_info(message):
        return True

    # 2. 包含重要决策或约定的
    if contains_decision(message):
        return True

    # 3. 用户明确要求记住的
    if "记住" in message or "记一下" in message:
        return True

    # 4. 纠正性信息（用户纠正了之前的错误）
    if context.get("is_correction"):
        return True

    return False
```

### 3.4 记忆检索策略

```
用户发送新消息
       │
       ▼
┌──────────────────┐
```

## 四、对话状态追踪（DST）

### 4.1 什么是 DST？

在多轮对话中追踪用户的意图、槽位（slot）和对话状态。

```
场景：订机票

轮次1:
  用户: "我想订一张去北京的机票"
  状态: {intent: "订票", destination: "北京", date: null, ...}

轮次2:
  用户: "下周三出发"
  状态: {intent: "订票", destination: "北京", date: "下周三", ...}

轮次3:
  用户: "从上海走"
  状态: {intent: "订票", destination: "北京", date: "下周三",
         origin: "上海", ...}
```

### 4.2 DST 实现方式

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 规则/模板 | 正则匹配+状态机 | 精确 | 不灵活 |
| LLM + Schema | LLM 按预定义 Schema 提取状态 | 灵活 | 需要结构化输出 |
| 端到端训练 | 专门训练 DST 模型 | 效率高 | 需要大量标注数据 |

**LLM + Schema 方式（推荐）**：

```python
state_schema = {
    "intent": "string | null",
    "entities": {
        "product": "string | null",
        "date": "string | null",
        "location": "string | null",
        "amount": "number | null"
    },
    "confirmed": "boolean",
    "missing_slots": ["string"]
}

# 每轮调用 LLM 更新状态
current_state = llm_extract_state(
    conversation_history=messages,
    schema=state_schema,
    previous_state=state
)
```

## 五、记忆系统设计——面试实战

> [!hint]- 面试场景：设计一个客服 AI 的记忆系统

> [!success]- 参考设计
> ```
> ┌─────────────────────────────────────────────┐
> │           客服 AI 记忆系统                    │
> │                                             │
> │ 第一层：会话记忆（短期）                      │
> │   - 当前对话的完整历史                       │
> │   - 最近的 10 轮交互                        │
> │   - 容量：~4K tokens                        │
> │                                             │
> │ 第二层：用户记忆（长期）                      │
> │   - 用户画像（会员等级、偏好、历史订单）      │
> │   - 存储在 Redis，实时更新                   │
> │   - 容量：~2K tokens                        │
> │                                             │
> │ 第三层：知识记忆（持久）                      │
> │   - 产品FAQ、政策文档                       │
> │   - 存储在向量数据库，按需检索               │
> │   - 容量：无限                               │
> │                                             │
> │ 工作流程：                                   │
> │   1. 用户消息 → 检索用户记忆 + 知识记忆      │
> │   2. 拼接 [用户画像] + [相关知识] + [对话历史]│
> │   3. LLM 生成回答                           │
> │   4. 更新会话记忆（+ 必要时更新用户记忆）     │
> │   5. 对话结束后 → 摘要存入长期记忆           │
> └─────────────────────────────────────────────┘
> ```

> [!hint]- 面试场景：记忆系统会不会越来越大，影响检索质量？

> [!success]- 记忆治理策略
> 1. **TTL（过期时间）**：临时记忆 24h 过期，重要记忆 30d 过期
> 2. **去重**：相同/相似的记忆只保留最新的
> 3. **合并**：多条细碎记忆合并为一条总结
> 4. **遗忘曲线**：越久没用过的记忆，优先级越低
> 5. **重要性评分**：LLM 给每条记忆打分，低分的定期清理
> 6. **分层存储**：热记忆（Redis）→ 温记忆（向量DB）→ 冷记忆（对象存储）

## 六、面试高频问题

> [!success]- Q1：LLM 的上下文窗口越来越大（128K-200K），还需要记忆系统吗？
> 需要！原因：
> 1. **成本**：128K tokens 的输入成本很高，不可能每次都传全部历史
> 2. **注意力稀释**：上下文太长，LLM 对早期内容的关注度下降（Lost in the Middle）
> 3. **跨会话**：上下文窗口无法跨会话持久化
> 4. **效率**：长上下文推理速度慢、延迟高
> 记忆系统的价值是**筛选出最相关的信息**，而不是装下所有信息。

> [!success]- Q2：如何评估记忆系统的效果？
> 1. **回指消解准确率**：用户说"那个问题"，系统能否正确解析
> 2. **信息保持率**：关键信息在长对话后是否仍然可用
> 3. **检索准确率**：检索到的记忆是否相关
> 4. **用户满意度**：用户对"被记住"的感知
> 5. **Token 效率**：用最少的 token 传达最多的记忆

> [!success]- Q3：多用户场景下，如何管理记忆隔离？
> 1. **用户 ID 隔离**：所有记忆存储带 user_id 标签，检索时过滤
> 2. **命名空间**：向量数据库中按 user_id 分 partition
> 3. **权限控制**：Agent 只能访问当前用户的记忆
> 4. **共享记忆**：部分知识（如产品FAQ）跨用户共享
> 5. **合规**：GDPR/隐私法规要求支持记忆删除

## 关键要点回顾

1. **记忆分类**：短期（对话历史）+ 长期（向量存储）+ 工作记忆（任务状态）
2. **上下文管理**：摘要 + 滑动窗口是最实用的策略
3. **MemGPT 思想**：LLM 自主管理记忆的读写，类似操作系统虚拟内存
4. **记忆治理**：TTL、去重、合并、遗忘曲线——防止记忆膨胀
5. **跨会话记忆**：对话结束后摘要存储，下次对话时检索注入
